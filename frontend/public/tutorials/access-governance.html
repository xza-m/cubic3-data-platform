权限治理闭环

权限体系应尽量薄：平台只管理成员、机器接入、权限包和可访问数据等级；MaxCompute 继续承担底层数据面的物理权限。管理员要能在一个页面完成配置、验证和回收。

18 min Admin 权限管理

核心概念

不要把权限拆成用户看不懂的多套体系。

1

成员

成员来自飞书 SSO 或内部引导账号。列表展示真实姓名，ID 只作为辅助识别，避免把 open_id 暴露给业务管理员。

2

机器接入

Bot、Skill、Job 使用机器主体和 API Key 认证。API Key 只证明调用方身份，不直接代表真人数据权限。

3

权限包

权限包把平台职责和数据等级收敛成少量可理解选项，例如语义建模、治理管理员、M0 公开语义、M1 聚合数据、M2 受控明细。

管理员配置流程

推荐按成员选择、配置、保存、验证四步完成。

1

选择成员

在成员列表按姓名、邮箱或来源筛选。右侧上下文只展示身份摘要，不在上下文面板里做复杂点选。

2

配置权限包

在主区域使用清晰的权限包选择器完成修改。保存前展示本次变更摘要，让管理员知道新增或移除了哪些能力。

3

验证数据访问

用 Agent preview 或查询预检验证 M0、M1、M2 行为。M3/raw 默认不直接开放，通过调整数据等级或数据策略来控制。

4

审计与回收

每次角色修改都记录操作者、目标成员、变更前后权限包和时间。离职或角色变化时可以快速回收。

权限产品闭环的判断标准：管理员能看懂成员是谁、能配置权限包、能验证查询结果、能在审计里追踪变更。
