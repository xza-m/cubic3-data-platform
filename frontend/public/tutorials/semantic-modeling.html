语义建模工作流

语义建模把“业务问题”沉淀为 Agent 可消费的 Cube、本体对象、指标和绑定关系。Copilot 负责生成草稿，建模人员负责确认字段、口径和发布范围。

25 min Intermediate 语义中心

推荐流程

从自然语言问题开始，但最终落到可校验、可发布的 spec。

1

描述业务问题

输入业务方真正想问的问题，例如“最近 7 天学生评论数按学校汇总”。系统会抽取主题、候选指标、候选维度和可能的源表。

2

编辑建模草稿

在工作台中确认 Cube 来源、维度表、指标 SQL、本体对象和业务指标绑定。不要把不确定口径直接发布给 Agent。

cube:
  name: student_comment_cube
  source: dwd_interaction_comment_reports_df
  measures:
    - name: comment_total_count
      sql: count(distinct comment_id)
ontology:
  business_metric: student_comment_total_count
  measure_ref: student_comment_cube.comment_total_count
3

校验并发布

发布前至少通过结构校验、指标绑定校验、Agent ready 校验。存在阻断项时，下一步必须给出可执行补齐方案，而不是只展示空建议。

质量标准

建模是否可用，关键看 Agent 是否能稳定解释和执行。

✓

字段含义可解释

每个维度、指标和筛选字段都有业务名、类型和必要说明。

✓

口径可追溯

指标公式、粒度、时间字段和物理表来源能在技术详情里完整追踪。

✓

发布后可消费

Cube 和 Ontology 均为 active，Data Agent 可以基于语义层回答问题。

建议先从一个高频业务问题建起最小可用语义，再逐步扩展维度和指标。不要一次性把整张宽表都包装成语义模型。
