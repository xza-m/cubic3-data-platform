## Context
本次变更不新增新的语义能力，也不重做全局导航与整体后台框架，而是在当前语义中心 workbench 基础上，继续收敛页面体验和任务闭环。

当前代码已经提供：

- 统一页级骨架：`SemanticPageShell`、`SemanticPageHeader`、`SemanticSurface`
- 统一状态与动作：`SemanticActionBar`、`SemanticStatusBanner`
- 统一侧边说明：`SemanticInspectorPanel`
- 已落地的核心路由与页面骨架：`CubeList`、`CubeStudio`、`DomainList`、`DomainCanvas`、`DevTools`

因此，本次设计重点不是“再造一个新后台”，而是基于现有实现解决以下问题：

- 页面首屏缺少明确 focus
- 同类信息在不同页面重复但不统一
- 工作列表、编辑流、画布流、调试流之间的边界已建立，但操作衔接还不够顺
- 共享 UI 模式尚未沉淀为语义中心内部复用模块

同时，本次变更已经补齐语义中心的设计语境。语义中心的核心用户为数据建模工程师、BI 开发和数据产品经理，核心任务是开发和治理 Cube、搭建领域模型、查指标定义。界面整体应体现“专业、分析、清晰”的品牌气质，参考 `Cube.js` 和 `dbt Cloud` 的现代数据栈工作台风格，但不复制其具体界面模式。

视觉方向上的明确约束包括：

- 保持浅色工作台基调
- 保留现代数据栈的结构感与工程感
- 可以比传统后台更轻、更有节奏，但不营销化
- 不采用毛玻璃卡片风
- 不做无谓的卡片堆叠
- 不退回到传统 CRUD 式列表预览
- 不将复杂任务页面做成表演型工作流

## Goals / Non-Goals
- Goals:
  - 统一语义中心的工作台体验基线
  - 明确语义中心的布局语言、视觉层级、状态语言与页面节奏
  - 让 Cube 管理页具备“问题优先”的处理视角
  - 让 Cube Studio 成为真正的分阶段编辑流程
  - 让领域目录具备治理视角，而不只是目录切换
  - 让领域建模页更像专业建模工作台，而不是组合式页面
  - 让 DevTools 更接近轻量 IDE 的上下文体验
  - 通过共享组件降低页面间重复实现
- Non-Goals:
  - 不修改全局顶部导航、左侧主导航、主路由结构
  - 不引入新的大型设计系统或替换现有 workbench 语言
  - 不在本次提案中增加新的后端领域规则
  - 不引入新的重量级图谱、布局或 IDE 依赖
  - 不将语义中心改造成营销风或大面积视觉重设计
  - 不用大 Hero、宣传型模块、装饰性指标卡替代真实工作台信息

## Visual Direction
### Design Thesis
语义中心不是“后台里的一组卡片页面”，而是“结构化语义工作台”。页面不依赖大面积装饰来建立记忆点，而依赖稳定的上下文页头、分析型状态条、内嵌式工作面板和精确的排版层级来建立识别度。

### Layout Language
- 普通管理页采用“上下文页头 + 状态条 + 主工作区 + 右侧检查器”
- 复杂工作台页采用“上下文页头 + 顶部操作带 + 左资源区 + 中央主画布/Workspace + 右侧 Inspector”
- 主任务区永远优先于辅助摘要区；检查器用于判断、说明、校验，不抢主编辑任务
- 页面首屏先回答三个问题：当前对象是什么、当前状态如何、下一步能做什么

### Visual Hierarchy
- 用排版、留白、细分隔和背景层次建立层级，不靠大卡片和夸张阴影
- `SemanticSurface` 区分为主体面板、辅助面板、内嵌工作面板三种语义层
- 大块营销式统计卡不进入语义中心主页面；首屏摘要以状态带、快筛和对象密度优先
- 圆角保持中小尺度，边框细而克制，阴影降到辅助级

### Color System
- 工作台保持浅色底，底色不使用纯白，建议中性色轻微偏冷
- 主强调色走冷静的蓝绿或冷青绿方向，用于“可发布、连接、当前焦点、活跃态”
- 风险色采用橙红体系，草稿态采用暖中性黄褐，不使用高饱和荧光色
- 避免大面积渐变、发光、玻璃态背景和品牌色泛滥

### Typography
- 标题承担对象识别与结构划分，不承载宣传语义
- 页标题、区块标题、辅助元信息、表格文本采用稳定的层级比例，而不是依赖图标和粗暴放大
- 代码、Schema、YAML、调试信息只在必要区域使用等宽字体，不把等宽字体扩散为整页风格

### State Language
统一状态表达体系：

- `Badge`：对象生命周期状态，如草稿、待发布、已发布、失败
- `Banner`：页面级阻塞、校验、发布前提醒
- `Rail`：列表行或节点的侧边状态轨，用于问题优先识别
- `Inline hint`：字段级、Join 级、Schema 级提示

统一状态优先级：

- 校验失败 / 阻塞
- 未绑定 / 缺失配置
- 草稿 / 待发布
- 正常已发布
- 高复用 / 高影响

### Motion
- 只保留关键反馈动效：状态切换、选中高亮、Inspector 切换、保存/发布反馈
- 避免为列表、卡片、图表添加装饰性动画
- 复杂区域优先用透明度和轻量位移做反馈，不做大幅缩放和弹跳

## Decisions
- Decision: 语义中心继续沿用现有 Semantic workbench 语言，而不是引入新的页面骨架
  - Why: 现有 workbench 已具备较强一致性，继续扩展符合 `KISS` 和 `DRY`
  - Alternatives considered:
    - 重新设计整套后台页模板
      - 缺点：与仓库现状割裂，改动面过大

- Decision: 语义中心采用“结构化工作台”而不是“卡片化后台”作为视觉主线
  - Why: 用户的核心任务是建模、治理、排查和定义确认，结构清晰与状态判断比装饰性更重要
  - Alternatives considered:
    - 保留大面积卡片与概览块
      - 缺点：会削弱对象密度与问题优先级，违背当前用户任务场景

- Decision: 通过页头、状态条、内嵌工作面板和检查器形成统一布局语言
  - Why: 这套模式能在不改全局导航的前提下形成语义中心内部的一致体验，符合 `KISS`
  - Alternatives considered:
    - 每个页面各自定义布局节奏
      - 缺点：页面间切换成本高，复用价值低，违背 `DRY`

- Decision: 用分析型状态语言取代装饰性视觉概览
  - Why: 语义中心最核心的是判断和处理优先级，而不是展示“看起来丰富”的总览模块
  - Alternatives considered:
    - 使用更多指标卡与彩色摘要块
      - 缺点：容易滑向营销型后台风格，且信息密度低

- Decision: Cube 管理以表格为主，预览为辅
  - Why: 企业级对象管理应优先支持筛选、排序和问题定位，而不是卡片浏览
  - Alternatives considered:
    - 延续双栏卡片浏览
      - 缺点：不利于高密度筛查和多维比较

- Decision: Cube Studio 采用单页分阶段工作流，而不是拆多路由向导
  - Why: 当前路由和状态模型已经稳定，多路由向导会增加保存、回退和兼容复杂度
  - Alternatives considered:
    - 拆成独立多步骤路由
      - 缺点：实现复杂、状态切换成本高，违背 `YAGNI`

- Decision: 领域建模页保持三栏，但通过层级重排突出画布中心
  - Why: 当前 React Flow + ELK 组合已经可用，问题在于信息权重，而不是技术能力不足
  - Alternatives considered:
    - 大幅抽屉化或单面板化
      - 缺点：会破坏专业建模场景下的并行观察能力

- Decision: DevTools 继续使用“资源树 + Workspace + Tabs”结构
  - Why: 当前结构已经具备 IDE 心智，优化重点应放在上下文和结果反馈
  - Alternatives considered:
    - 合并为多页面工具导航
      - 缺点：打断当前对象上下文，降低连续性

- Decision: 新增组件只在语义中心内部沉淀，不扩散为全站设计系统
  - Why: 当前目标是加速语义中心迭代，而不是扩大抽象层
  - Alternatives considered:
    - 立即建立全站统一设计系统
      - 缺点：超出当前需求范围，违背 `YAGNI`

## Page-Level Layout Guidance
### Cube-Driven Standard Pattern
`Cube 管理页` 已经验证出一套更适合语义中心的页面标准，后续语义页默认遵循该模式，而不是各页继续发明自己的开场方式。

标准层级如下：

- `轻页头`
  - 只保留页面标题、一句 utility copy、1 个主动作和可选 1 个次动作
  - 不再在页头堆叠多枚 badge、说明胶囊和英文 eyebrow
- `单条上下文`
  - 页头下方统一使用 1 条 `SemanticWorkbenchContextBar`
  - 只表达当前范围、当前状态、规模信息和当前透镜，不重复解释页面职责
- `主工作面板`
  - 页面主体只保留 1 个主工作面板或 1 组主工作台区域
  - 过滤、列表、资源树、画布、步骤轨、Workspace 都属于主工作面板内部，不再额外套“说明区”
- `条件显示 Inspector`
  - Inspector、预览和摘要面板只在当前任务需要时显示
  - 列表页默认不因为空选中而长期占据主版面

标准约束如下：

- 管理页不再使用大面积统计卡作为首屏主视觉
- 同一页内不重复表达“当前状态 / 当前对象 / 下一步动作”
- 英文眉标题、英文区块名和解释性短句默认移除，除非是技术对象本身名称
- 复杂工作台页可以保留状态条，但状态条只用于阻塞和发布前判断，不再承担页面导览

### CubeList
- 页面采用“页头 + 快筛状态带 + 主表格 + 右侧预览检查器”
- 主表格承载对象身份、状态、绑定、复用、更新时间、负责人等判断信息
- 右侧检查器用于当前对象摘要和下一步动作，不退回传统详情页

### CubeStudio
- 页面采用“步骤轨 + 当前步骤工作区 + Inspector”
- 当前步骤成为唯一视觉主任务，其余步骤折叠为摘要
- `Schema`、字段结构、规则预览使用更内嵌的工作面板语义

### DomainCanvas
- 页面采用“顶部建模上下文条 + 左资源库 + 中央画布 + 右 Inspector”
- 中央画布是绝对主角，左右侧栏使用更轻背景与更高信息密度
- 节点、连线、冲突、Join 缺失通过统一状态语言表达，而不是靠装饰

### DevTools
- 页面采用“上下文条 + 左资源树 + 中 Workspace + 结果/同步 Tabs”
- 编辑器与编译结果区是内嵌工作面板，不采用营销式卡片或层层包裹
- 对象上下文、最近执行结果、同步状态必须在首屏可扫描

### Standard Rollout
- `Overview`：改为“轻页头 + 上下文条 + 模块现状主面板”，不再拆成多块解释页
- `DomainList`：保留目录 rail 和治理列表，但移除状态 banner，改为一条上下文条承载当前目录、透镜和分页状态
- `DomainModelingEntry`：保留“创建 + 继续”双栏结构，但收回英文眉标题和 badge 云，统一为上下文条 + 单主面板
- `DevTools`：把当前对象、标签页、Schema 状态收回到页头下方单条上下文，Workspace 内不再重复堆第二层上下文
- `CubeStudio` / `DomainCanvas`：保留原有工作流和状态条，仅补齐统一上下文条与更轻的页头节奏

## Risks / Trade-offs
- 页面拆分后短期内会有较多组件新增
  - Mitigation: 限定在 `components/Semantic/` 下沉淀，避免过早全站化

- 加入明确视觉语言后，局部页面可能与主应用其它旧页面产生风格差异
  - Mitigation: 将本次工作限制在语义中心范围内，并尽量复用现有 `Semantic workbench` token 和组件

- Cube 管理改表格视图后，当前用户对卡片浏览的习惯会变化
  - Mitigation: 保留右侧预览，避免完全失去对象摘要视图

- 领域建模页继续保留三栏，桌面横向空间仍需精细控制
  - Mitigation: 控制左右栏宽度，必要时引入折叠和过滤，而不是彻底改布局

- DevTools 资源树增强后，状态管理复杂度会上升
  - Mitigation: 继续用 `useUrlState`，不额外引入新状态库

## Migration Plan
1. 先定义共享 workbench 模式和语义中心 UI 复用组件
2. 优先重构 `CubeList`，建立列表、筛选、预览的统一模式
3. 再重构 `CubeStudio`，完成单对象编辑流的阶段化
4. 随后增强 `DomainList` 与 `DomainCanvas`，打通目录治理和建模回流
5. 最后优化 `DevTools`，统一资源树和 Workspace 上下文体验
6. 补充对应验收与页面级验证
7. 在实现过程中同步沉淀页面级布局、状态和视觉 token 的复用规则
