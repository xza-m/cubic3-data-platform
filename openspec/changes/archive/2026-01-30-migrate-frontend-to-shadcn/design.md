# Design: Frontend Framework Migration to shadcn/ui

## Context

**Current State:**
- Frontend built with React 18.2 + Ant Design 5.13.2
- 51 files importing from 'antd'
- 34 files using complex components (Table, Modal, Drawer, etc.)
- Tailwind CSS 3.4.1 already configured
- Bundle size: ~2MB (largely due to Ant Design)

**Stakeholders:**
- Frontend developers (daily development impact)
- End users (UI/UX changes, performance improvements)
- Product team (design control and consistency)

**Constraints:**
- Must maintain all existing functionality
- Cannot break user workflows during migration
- Timeline: 1-2 weeks for complete migration
- Must be testable incrementally

## Goals / Non-Goals

**Goals:**
- Reduce bundle size by at least 70% (~2MB → ~300KB)
- Achieve complete design customization capability
- Modernize UI aesthetics with glassmorphism and contemporary styling
- Improve component consistency across the platform
- Maintain or improve component accessibility
- Enable faster iteration on UI changes

**Non-Goals:**
- Redesigning user workflows or page layouts (structure remains the same)
- Changing backend APIs or data contracts
- Adding new features during migration (pure refactor)
- Supporting both Ant Design and shadcn/ui simultaneously (complete switch)

## Decisions

### Decision 1: Use shadcn/ui (not pure Headless UI)

**Rationale:**
- shadcn/ui provides 50+ pre-built components with excellent UX
- Built on Radix UI (accessible primitives) + Tailwind CSS
- Components are copied into project (full ownership and customization)
- Active community and extensive documentation
- Project already has Tailwind CSS configured

**Alternatives Considered:**
- **Pure Headless UI**: Rejected - too much work to build all components from scratch (80-120h)
- **Keeping Ant Design with theme customization**: Rejected - still large bundle, limited design freedom
- **Hybrid approach (Ant Design for complex components)**: Rejected - confusing DX, still large bundle

### Decision 2: Complete migration (not gradual)

**Rationale:**
- Cleaner codebase without mixed component libraries
- Simpler mental model for developers
- Eliminates Ant Design entirely, maximizing bundle size reduction
- Avoids long-term maintenance of two UI libraries

**Risk Mitigation:**
- Work in feature branch with comprehensive testing
- Deploy to test environment before production
- Keep main branch stable for rollback

### Decision 3: React Hook Form for form management

**Rationale:**
- Industry standard for React forms
- Better TypeScript support than Ant Design Form
- Smaller bundle size and better performance
- Integrates well with shadcn/ui Form components

**Migration Path:**
- Replace `Form` → `<form>` + `useForm` hook
- Replace `Form.Item` → `<FormField>` + `<FormItem>`
- Replace `rules` prop → Zod schema validation
- Replace `form.validateFields()` → `handleSubmit()`

### Decision 4: Rebuild complex components (Table, Upload)

**Component Strategy:**

| Component | Approach | Justification |
|-----------|----------|---------------|
| Table | shadcn/ui DataTable | Feature-complete example available, supports sorting/filtering/pagination |
| Modal | shadcn/ui Dialog | Direct replacement, similar API |
| Drawer | shadcn/ui Sheet | Direct replacement |
| Form | React Hook Form + shadcn/ui Form | Better DX, smaller bundle |
| Select | shadcn/ui Select | Direct replacement |
| DatePicker | shadcn/ui Calendar + Popover | Requires minor adapter logic |
| Upload | Custom with react-dropzone | No direct shadcn equivalent, but simple to build |

**Not Rebuilding:**
- Monaco Editor (standalone, no Ant Design dependency)
- Lucide Icons (already used, no change needed)

### Decision 5: Two-layer component architecture

**Structure:**
```
frontend/src/components/
├── ui/              # shadcn/ui raw components (auto-generated by CLI)
│   ├── button.tsx
│   ├── select.tsx
│   └── ...
└── business/        # Business wrapper components
    ├── FormSelect.tsx      # Pre-configured Select for forms
    ├── DataTable/          # Enhanced table with common patterns
    ├── PageCard.tsx        # Standard page card layout
    └── ...
```

**Rationale:**
- **ui/**: Generated by shadcn CLI, minimal customization, easy to update
- **business/**: App-specific wrappers, consistent styling, reusable patterns
- Clear separation of concerns
- Easy to update shadcn components without affecting business logic

## Technical Approach

### Phase 1: Infrastructure Setup (Day 1)

```bash
# Install shadcn/ui
npx shadcn@latest init

# Install core components
npx shadcn@latest add button input select ...
```

**Configuration Changes:**
- `tailwind.config.js`: Set `preflight: true`, add shadcn theme tokens
- `package.json`: Add `@radix-ui/*`, `class-variance-authority`, `tailwind-merge`
- `tsconfig.json`: Add `@/components` alias
- `index.css`: Remove Ant Design overrides, add shadcn CSS variables

### Phase 2: Component Library (Days 2-3)

**Create Business Wrappers:**

```typescript
// FormSelect.tsx - Consistent select component
export function FormSelect({ options, ...props }: FormSelectProps) {
  return (
    <Select value={props.value} onValueChange={props.onChange}>
      <SelectTrigger className={props.className}>
        <SelectValue placeholder={props.placeholder} />
      </SelectTrigger>
      <SelectContent>
        {options.map(opt => (
          <SelectItem key={opt.value} value={opt.value}>
            {opt.label}
          </SelectItem>
        ))}
      </SelectContent>
    </Select>
  )
}
```

**DataTable Component:**

Based on shadcn/ui DataTable example, with additions:
- Server-side pagination
- Column sorting and filtering
- Row selection
- Bulk actions
- Loading and empty states
- Responsive mobile view

### Phase 3: Page Migration (Days 4-9)

**Migration Strategy:**

| Priority | Pages | Complexity | Estimated Time |
|----------|-------|------------|----------------|
| High | Dashboard, AppMarket, Channels | Simple (cards, lists) | 6h |
| Medium | Datasources, Datasets, ExecutionMonitor | Table + filters | 12h |
| High | ExtractionTasks, Editor, DatasetDetail | Multi-step forms, tabs | 12h |

**Migration Pattern:**

```typescript
// Before (Ant Design)
import { Select, Button, Table } from 'antd'

<Select>
  <Select.Option value="1">Option 1</Select.Option>
</Select>

// After (shadcn/ui)
import { FormSelect } from '@/components/business'

<FormSelect 
  options={[{ label: 'Option 1', value: '1' }]}
  value={value}
  onChange={setValue}
/>
```

## Risks / Trade-offs

### Risk 1: Component Feature Parity

**Risk:** shadcn/ui components may lack advanced features of Ant Design

**Mitigation:**
- Audit Ant Design component usage before migration
- Identify critical features (e.g., Table column resizing)
- Build missing features or find acceptable alternatives
- Document any feature drops and get product approval

**Trade-off:** Some advanced features may require custom implementation, but gain full control

### Risk 2: Form Migration Complexity

**Risk:** Migrating from Ant Design Form to React Hook Form is not 1:1

**Mitigation:**
- Create migration guide for common patterns
- Build helper functions for common validations
- Provide code examples for typical form structures
- Reserve extra time for complex forms (ExtractionTaskConfig)

**Trade-off:** Short-term migration effort for long-term better DX

### Risk 3: Third-Party Library Dependencies

**Risk:** `@rjsf/antd` (JSON Schema Form) tightly coupled to Ant Design

**Options:**
1. Keep `@rjsf/antd` isolated (temporary compromise)
2. Migrate to `react-hook-form` with manual schema handling
3. Use `@rjsf/core` with custom shadcn theme

**Decision:** Option 2 - Full migration for consistency

**Trade-off:** More migration work, but cleaner architecture

### Risk 4: Timeline Overrun

**Risk:** 2-week timeline may be insufficient

**Mitigation:**
- Prioritize core pages first (Data Center, Extraction Center)
- Keep non-essential pages on Ant Design temporarily if needed
- Have rollback plan ready
- Daily progress tracking

**Contingency:** Extend to 3 weeks if needed, or phase rollout (core pages first)

### Risk 5: Visual Regression

**Risk:** UI changes may confuse existing users

**Mitigation:**
- Maintain similar layouts and workflows
- Progressive disclosure of changes (test environment first)
- User communication about UI refresh
- Collect feedback during test phase

**Trade-off:** Temporary learning curve for improved long-term UX

### Rollback Procedure

If critical issues arise:

1. **Immediate:** Switch deployment back to `main` branch
2. **Investigation:** Identify blocking issue
3. **Decision:**
   - If fixable in <2 days: Fix in feature branch
   - If longer: Defer migration, revisit plan
4. **Communication:** Inform stakeholders of status

## Open Questions

1. **JSON Schema Form Strategy:** Should we keep `@rjsf/antd` temporarily or migrate immediately?
   - **Recommendation:** Migrate to React Hook Form for consistency

2. **Storybook Setup:** Should we set up Storybook for component development and testing?
   - **Recommendation:** Nice to have, but not critical for migration (defer to post-migration)

3. **Design Tokens:** Should we formalize a design token system beyond Tailwind config?
   - **Recommendation:** Start with Tailwind config, formalize later if needed

4. **Testing Strategy:** Do we need visual regression testing tools (Percy, Chromatic)?
   - **Recommendation:** Manual visual testing is sufficient for this migration

5. **Gradual Rollout:** Should we roll out to users gradually (feature flag)?
   - **Recommendation:** Test environment first, then full prod deployment (no gradual rollout)
