# Change: 配置中心前端界面

## Why

### 当前问题

配置中心（订阅中心）的后端 API 已完成实现，但**缺少前端管理界面**：

1. **无可视化管理**：用户无法通过界面管理推送渠道和订阅规则
2. **依赖 API 调用**：必须通过 curl/Postman 进行配置操作
3. **缺少入口**：侧边栏没有配置中心的菜单项

### 现有后端能力

| API 端点 | 功能 |
|----------|------|
| `GET/POST /api/v1/channels` | 渠道列表和创建 |
| `GET/PUT/DELETE /api/v1/channels/{id}` | 渠道详情和管理 |
| `GET/POST /api/v1/subscriptions` | 订阅列表和创建 |
| `GET/PUT/DELETE /api/v1/subscriptions/{id}` | 订阅详情和管理 |
| `GET /api/v1/app-instances/{id}/subscriptions` | 应用实例订阅查询 |

## What Changes

### 核心变更

1. **侧边栏菜单**
   - 在"应用中心"分组下添加"配置中心"入口
   - 包含"渠道管理"和"订阅管理"两个子菜单

2. **渠道管理页面** (`/config/channels`)
   - 渠道列表（表格展示）
   - 创建/编辑渠道表单（Modal）
   - 渠道类型选择（飞书/Webhook/邮件/OSS）
   - 动态配置表单（根据渠道类型显示不同字段）

3. **订阅管理页面** (`/config/subscriptions`)
   - 订阅列表（表格展示）
   - 创建/编辑订阅表单（Modal）
   - 应用实例选择器
   - 渠道选择器
   - 事件类型多选
   - 过滤条件配置

4. **API 客户端**
   - 新增 `frontend/src/api/channels.ts`
   - 新增 `frontend/src/api/subscriptions.ts`

## Design Decisions

### 1. 页面布局

**决策**：采用与现有页面一致的 Glassmorphism 风格

- 使用现有的卡片、表格、表单组件
- 保持视觉一致性
- 复用现有的 CSS 类

### 2. 渠道配置表单

**决策**：根据渠道类型动态渲染配置字段

| 渠道类型 | 配置字段 |
|----------|----------|
| `feishu` | chat_id, webhook_url (可选) |
| `webhook` | url, method, headers, secret |
| `email` | recipients, smtp_config (后续) |
| `oss` | bucket, path_template (后续) |

**实现方式**：
- 使用 `@rjsf/antd` (React JSON Schema Form) 动态生成表单
- 后端 `Channel.get_config_schema()` 返回 JSON Schema

### 3. 订阅事件类型

**决策**：提供预定义的事件类型列表

```typescript
const EVENT_TYPES = [
  { value: 'app.execution.completed', label: '应用执行完成' },
  { value: 'app.execution.failed', label: '应用执行失败' },
  { value: 'app.instance.created', label: '应用实例创建' },
  { value: 'app.instance.enabled', label: '应用实例启用' },
  { value: 'app.instance.disabled', label: '应用实例禁用' },
]
```

### 4. 过滤条件配置

**决策**：支持简单的 Key-Value 过滤器

- 用户可添加多个过滤规则
- 支持 `app_code`, `status`, `triggered_by` 等字段
- 使用动态表单添加/删除规则

### 5. 用户体验

**决策**：提供快捷操作

- 渠道列表支持快速启用/禁用
- 订阅列表支持按应用/渠道筛选
- 创建订阅时可从应用详情页快速跳转

## Non-Goals

本次变更**不包括**：

- ❌ 渠道消息预览功能
- ❌ 订阅触发历史记录页面（后续需求）
- ❌ 批量导入/导出功能
- ❌ 复杂的过滤条件构建器（仅支持简单 Key-Value）

## Risks

### 风险 1：表单复杂度

**描述**：不同渠道类型的配置表单差异大，维护成本高。

**缓解措施**：
- 使用 JSON Schema 驱动的动态表单
- 复用后端 `get_config_schema()` 返回的 Schema

### 风险 2：应用实例选择器性能

**描述**：应用实例数量多时，下拉选择器可能卡顿。

**缓解措施**：
- 使用虚拟滚动
- 支持搜索过滤
- 分页加载
