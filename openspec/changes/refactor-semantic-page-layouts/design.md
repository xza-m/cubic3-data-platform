## Context
这次变更不重新讨论语义中心的功能页划分，而是在既有边界下重构页面内部布局和功能点组织。

本次默认接受以下四类页面边界：

- `Cube 管理`：查看、检索 Cube 的维度和指标
- `Cube 设计`：新建或编辑单个 Cube 的维度、指标和基础属性
- `领域管理`：查看、检索领域目录和领域内关联概况
- `领域设计`：新建领域，在画布中纳入 Cube、建立关系并发布

问题本质是页面内部没有按“用户任务”组织，而是按“技术实现 + 对象信息 + 可跳转能力”混排。

## Goals / Non-Goals
- Goals:
  - 让四个页面都在首屏表达单一主任务
  - 让管理页只承担“查找和概览”，设计页只承担“编辑和发布”
  - 删除无效跨页跳转和重复摘要
  - 形成页面职责级的测试闭环，而不只验证接口和主流程可走通
- Non-Goals:
  - 不修改后端语义模型、目录模型和查询语义
  - 不新增新的一级/二级模块
  - 不将 `IDE` / `DevTools` 提升为业务主路径
  - 不在本次变更中引入新的 DSL、查询 API 或物化策略能力

## Decisions
- Decision: 页面内部设计以“单页单主任务”为最高约束
  - Why: 当前主要认知负担来自页面内同时承载浏览、编辑、创建、跳转和说明
  - Alternatives considered:
    - 继续通过说明卡解释复杂页面
      - 缺点：无法真正降低认知负担，只会继续堆叠信息

- Decision: 管理页与设计页严格分工
  - Why: 管理页的职责是查找和理解对象，设计页的职责是修改对象；两者混排会直接破坏心智
  - Alternatives considered:
    - 在管理页内保留大量编辑能力
      - 缺点：会使页面同时承担列表、表单和流程控制，违背单一职责

- Decision: 跳转只保留顺主流程动作
  - Why: 当前无效跳转会打断用户对页面边界的理解
  - Alternatives considered:
    - 保留所有跨工作区捷径
      - 缺点：虽然“信息完整”，但会让业务页变成跳转台

- Decision: 说明、帮助和诊断信息后置
  - Why: 首屏必须优先展示上下文和主操作，帮助内容不应抢占主要视觉区域
  - Alternatives considered:
    - 在首屏展示完整说明
      - 缺点：业务用户第一眼无法判断当前能做什么

- Decision: 测试矩阵要覆盖“布局职责”而不仅是“流程成功”
  - Why: 当前页面的问题主要是职责混排；仅验证流程通过无法防止页面继续失控
  - Alternatives considered:
    - 只保留现有 smoke
      - 缺点：无法约束首屏主任务、重复摘要和无效跳转等交互回归

## Per-Page Design Contract
### Cube 管理
- 主任务：找到 Cube，理解它包含哪些维度和指标
- 首屏区域：
  - 页头：标题、副标题、`新建 Cube`
  - 筛选条：搜索、状态、所属领域、类型
  - 主区：左侧列表，右侧当前 Cube 预览
- 允许动作：
  - 搜索/筛选
  - 选择 Cube
  - 进入 `Cube 设计`
- 禁止动作：
  - 快速查询
  - DSL 编辑
  - 跳往领域设计和技术工作区

### Cube 设计
- 主任务：定义单个 Cube 的基础信息、维度和指标
- 首屏区域：
  - 页头：返回、名称、状态、保存动作
  - 主区：`基础信息 / 维度 / 指标`
  - 次区：校验面板
- 允许动作：
  - 编辑维度、指标、基础信息
  - 保存、激活、弃用
- 禁止动作：
  - 领域关系编辑
  - 查询验证
  - 目录浏览

### 领域管理
- 主任务：按目录查找领域，查看领域内关联概况
- 首屏区域：
  - 页头：标题、副标题、`新建领域`
  - 主区：左侧 Catalog 树，中间领域列表，右侧领域概览
- 允许动作：
  - 选择目录
  - 搜索领域
  - 查看领域状态、Cube 数、Join 数
  - 进入 `领域设计`
- 禁止动作：
  - 大表单编辑当前领域
  - 在首页同时新建另一个领域草稿
  - 发布阻塞详情抢占主区

### 领域设计
- 主任务：在画布中纳入 Cube、建立 Join、发布领域
- 首屏区域：
  - 页头：返回、领域名称、状态、阻塞项、保存/发布
  - 主区：左侧 Cube 库，中间画布，右侧 Inspector
- 允许动作：
  - 新建领域
  - 拖入 Cube
  - 编辑 Join
  - 保存草稿和发布
- 禁止动作：
  - 目录台账
  - 查询构建器
  - YAML 编辑器
  - 另一个领域的创建表单

## Risks / Trade-offs
- 页面重排可能会影响已有自动化选择器
  - Mitigation: 在重排前先冻结新的 `data-testid` 契约，并同步更新 E2E

- 管理页剥离编辑能力后，部分用户可能觉得“少了一个快捷入口”
  - Mitigation: 保留顺主流程的一步跳转，例如 `进入设计`，而不是完全断开

- 如果不适度保留公共组件，四个页面可能再次出现样式分叉
  - Mitigation: 只抽取最小共享骨架和摘要组件，不在本次前置抽过重壳层

## Migration Plan
1. 先冻结四个页面的职责和禁止动作清单
2. 优先重构 `领域管理`，清除“列表 + 表单 + 新建草稿”混排
3. 收敛 `领域设计` 为纯画布编排页
4. 收敛 `Cube 管理` 为台账 + 预览页，移出快速查询
5. 收敛 `Cube 设计` 为单 Cube 定义页
6. 最后统一补组件单测、页面交互测试、E2E 和视觉回归
