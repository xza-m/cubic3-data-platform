## Phase Boundary

**目标**：Phase 8 把 DataChat 切到 `runtime_mode="official"` 后，问数"符合 L1 理解意图 → 绑 L2 已发布资产 → L2 编译执行"这条理想架构仍有三处偏离。本期（8.1）收口**前两处**——治理地基对齐 + 两条物理直表旁路彻底删除——并把"答不出"统一收敛到诚实兜底。L1 意图理解升级（第三处偏离）留 **Phase 8.2**。

**本期交付（阶段 0-2）：**
1. **治理管线对齐**：DataChat 主链与全局 Agent API 走同一治理裁决（`AccessPolicyDecisionService` 的 `pre_route`/`post_compile`），不再写死 `viewer_roles=[]`。
2. **principal 透传 + 拒匿名**：send-message 必须登录，principal 解析在 interfaces 层经 Command 透传。
3. **两条物理直表旁路彻底删除**：legacy 第 3 层 + agent 第 2 层。
4. **统一诚实兜底**：`_build_unanswerable_fallback()`，`source='fallback'`/`status='unanswerable'`。
5. **合流收尾**：前端 dataset 文案校准为"范围提示（可选）"；跨入口治理一致性冒烟；回归与文档同步。

**本期不做（Out of scope）：**
- L1 意图理解升级（grounding 白名单 / LLM 抽取 / eval）→ Phase 8.2。
- `dataset_id` 真正下推为 router scope 过滤（默认仅范围提示）→ deferred。
- 执行端收敛（DataChat `_runtime_service.execute` vs gateway）→ deferred，本期保守不强收敛。

## 关键背景：三处偏离 + 两条物理泄漏（深挖纠正）

深挖发现原始"三点"框定有盲区：
- **物理直表有两条路，不是一条**：① legacy 第 3 层 `_handle_via_legacy_llm`→`_execute_query`→`AdapterFactory`（绕 dw-query-gateway）；② **agent 第 2 层** `DataChatChannel.to_agent_request`（取物理 schema + 建物理 adapter）→ agent loop `tool_registry.py:144 self._adapter.execute_query`。两条都违反 `base_adapter.py:117`"正式 Agent 查询必须经 dw-query-gateway"。
- **DataChat 治理是另一条管线**：DataChat 主链 `execute_plan` 全文零处 `access_policy`/`pre_route`/`post_compile`；Agent API 经 `AgentPlanHandler` 跑了这些裁决。只注入 principal 只让"角色非空"，不产生 allow/deny——必须让 DataChat 真正经过 `AccessPolicyDecisionService`。
- **`runtime_mode='official'` 已落地**（Phase 8 完成，`send_message_handler.py`），本期无此前置阻塞。

<decisions>
## Implementation Decisions（用户 2026-06-26 锁定，全选更严格项）

### 决策 1：治理对齐 = 完整对齐（不是只透传 principal）
- DataChat 主链（`_handle_via_semantic_router`）必须真正经过 `AccessPolicyDecisionService` 的 `pre_route` → (命中) plan/compile → `post_compile`，或直接委托既有 `AgentPlanHandler.handle` 拿到带 `policy_decision` 的结果再落消息。
- 验收：同一治理主体在 DataChat 与 `/agent/semantic/plan` 提同一问，`policy_decision.decision` 一致。
- **复用** `AccessPolicyDecisionService`（DI 已有 `container.py:323`），零新建。

### 决策 2：两条物理直表路 = 彻底删除（无 env 灰度回退口子）
- 删 legacy 第 3 层物理分支 + `_execute_query` + `LEGACY_DISCLAIMER`。
- 删/改 agent 第 2 层物理 adapter 直查（`DataChatChannel` + `tool_registry.py:144`）：全局会话不走物理 adapter。
- 不保留 `DATACHAT_LEGACY_PHYSICAL_SQL_ENABLED` 之类的灰度开关（对抗审查结论：留回退口子本身是反模式，违反 `base_adapter` 契约 + L1/L2 纪律）。

### 决策 3：匿名基线 = 必须登录
- `conversations.py` send-message 从 `optional_auth` 收紧为拒绝匿名（与 `agent.py` `@require_auth` 同口径）。
- 前端 DataChat 对未登录用户引导登录。

### 决策 4：principal 注入瘦身（对抗审查）
- 不在 handler 注入 `PrincipalResolver`——`execute_plan` 内部已 `_resolve_roles` 从 `principal_context['roles']` 取角色；handler 只把 `command.principal_context`/`viewer_roles` 透传即可。
- principal 解析（碰 Flask `g`）只在 interfaces 层（`conversations.py`），经 `SendMessageCommand` 新增可选字段透传进 application。

### 决策 5：统一诚实兜底 + agent 全局会话短路
- 三类"答不出"（semantic 软失败 / 未命中 / agent 软失败）统一收敛到 `_build_unanswerable_fallback()`，`source='fallback'`/`status='unanswerable'`，不产 SQL、不碰物理表。
- `_handle_via_agent` 对 `dataset_id is None` 的全局会话 `return None` 短路，消除"建 adapter→抛错被吞→残留 running AgentQueryLog"死路径。

### 🔴 硬约束（来自 ROADMAP M7 完成状态的遗留项，本期必须一并处理）
**完整治理对齐有"全 deny → 答不出"塌缩风险**：ROADMAP 标注 published cube 经 DataChat 消费被治理访问层默认拒绝（`governance/access.py:698` 无匹配访问策略→deny；`observe` 模式在 router 路径仍阻断）。若只接管线 + 传真实角色而不配访问策略，完整对齐会让问数**全部 deny → 全走诚实兜底（答不出）**。本期治理对齐**必须同时**：
1. handler 传真实角色（已在决策 1/4）；
2. 走 `AccessPolicyDecisionService` pre_route/post_compile（已在决策 1）；
3. **给已发布 cube 配默认访问策略**（否则全 deny）；
4. **查清 `observe` 模式为何在 router 路径仍阻断**并修正（`observe` 应放行 + 写审计，不阻断）。
> 这是"治理对齐能真正出数"而非"对齐后全部答不出"的关键。规划阶段须把这 4 项作为同一原子批次。

### 🔧 对账修订（2026-06-26，DB + 代码实测，覆盖上面 4 项硬约束 —— 以本节为准）

> 二次深挖 workflow + DB 实查后，上面 4 项硬约束有 3 项被实测纠正，**大幅瘦身**：

1. **决策1「加治理管线」→ 瘦身为「只补 principal 透传」**：实测 `app/application/execution_compiler/runtime_service.py:113` 已调 `AccessPolicyDecisionService.post_compile(...)`，`:126` `decision!=allow → blocked`。**DataChat 的 `execute_plan`→`runtime_service` 链路早就走了 post_compile 治理裁决并强制 deny**——原说"execute_plan 全文零处 access_policy"只对 handler/preview_service 文件本身成立，整链已治理。**不要重复引入 post_compile**；`pre_route` 实测 router 路径未调（grep 空），属可选早拦防御（post_compile 已兜底），本期可不补或低优先。核心只剩：principal 透传（决策4）。
2. **硬约束③「配默认访问策略」→ 删除（已满足）**：DB 实查 `access_data_policies` 已 seed 且 active（`m0_public_read`/`m1_aggregate_read` allow p20/`m2_detail_read`/`m3_raw_block` deny）；`access_execution_profiles.mc_m1_reader` active。答题 cube `dws_`→M1→`m1_aggregate_read` 结构上已覆盖。**不需新策略/种子**。
3. **硬约束④「修 observe」→ 删除（observe 不动）**：observe 只治理 row-scope（`access.py:749`），access-grant 段（:665-714）全程不读 mode、始终强制。本案 deny 在 access-grant 层（`data_policy_not_matched`），与 observe 无关。**保持 `RLS_ENFORCEMENT_MODE=observe`**。
4. **真正命门（DB 坐实）= 角色未到治理引擎**：`access_role_bindings` 实查——飞书用户 `principal:feishu:140b0c11...` 已持 `data_m0/m1/m2_reader`（透传后直接 allow 出数）；`principal:internal:local:admin` **零 data_ 角色**→匹配不上→deny。且治理 `PrincipalResolver.resolve` 只读 `principal_context.roles`、丢弃 `viewer_roles`。所以本期"真出数"只需：(a) handler 透传 `command.principal_context`（决策4，不止 viewer_roles）；(b) 验证用**有 data_ 角色的 principal**（飞书用户现成；internal admin 缺，验证时 `PUT /access/principals/<id>/permission-packages {data_m1_reader}` 授一下，非代码任务）。
> 结论：「全 deny 塌缩」被夸大——策略/档/角色体系都齐，缺的只是 principal 透传 + 消费者持 data_ 角色。本期原子批次实际只剩：**principal 透传（决策4）+ 删两条物理直表旁路（决策2）+ 拒匿名（决策3）+ 统一诚实兜底（决策5）**；硬约束③④移除，决策1 瘦身为不加冗余 post_compile。验收用飞书用户口径问"学生答题统计 总数"应 route_type=cube 且出数。

### Claude's Discretion（实现细节）

- 受影响测试的改写细节。
