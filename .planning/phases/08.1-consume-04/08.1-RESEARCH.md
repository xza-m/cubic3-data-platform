## Summary

本期是**纯内部架构收口**，零新外部依赖、零新框架/端口/枚举，全部复用既有构件。研究对 CONTEXT 全部锚点逐行核实，结论与记忆深挖一致但**行号有偏移**（已全部用当前代码行号纠正，例如 `AccessPolicyDecisionService` DI 在 `container.py:907` 而非记忆里的 `:323`；`tool_registry` 在 `app/application/agent/services/tool_registry.py` 而非 `app/domain/agent/`）。

**最高风险（Q1 全 deny 塌缩）已坐实根因并给出可执行最小方案**：`access.py:693-714` 的 deny 是 `data_policy_not_matched`——当主体角色与 active policy 的 `subject_roles` 无交集时，`DataPolicy.matches()`（`entities.py:79`）直接 `return False`，导致无 allow policy 命中 → deny。答题 cube 物理表 `dws_study_student_answer_kb_stat_di`（`dws_` 前缀）被判 **M1**（`access.py:264`），M1 的 allow policy `m1_aggregate_read`（`seed.py:602`）要求主体持有 `data_m1_reader`。当前 handler 写死 `viewer_roles=[]` → 主体无 data 角色 → 必 deny。**`observe` 模式无法救这一步**：observe 只在 `_post_compile_with_repository` 的 **row_scope 求值阶段**（`access.py:749-802`）才有"不阻断"语义；"无匹配 allow policy"的 deny（`:693`）发生在 row_scope 之前，与 RLS 模式无关。这就是"observe 仍阻断"的真相——不是 observe 没生效，而是 deny 根本不在 observe 的作用域内。

**最小放行方案**：把 DataChat 登录主体绑定到 `data_m1_reader`（+ 视答题表层级可能需 `data_m0_reader`，见 `catalog.py:96-99` 权限包 `data_m1_reader` 已含 `data_m0_reader`）。这比改 policy 的 `subject_roles=[]`（全放行，破坏治理语义）更克制，符合"完整治理对齐能真正出数"的口径。规划须把"配主体角色绑定 + handler 传真实角色 + 走 pre_route/post_compile"作为同一原子批次（CONTEXT 硬约束已要求）。

**Primary recommendation:** 治理对齐用**方案 B（handler 内调 `AccessPolicyDecisionService.pre_route/post_compile`，编译产物从 `execute_plan` 的 plan 提取）**而非方案 A（整体委托 `AgentPlanHandler`）——方案 A 会丢掉 DataChat 现有的消息落库/trace/AgentQueryLog 结构（`send_message_handler.py:111-173`），改动面反而大。物理收口用**全局会话短路**（`_handle_via_agent` 对 `dataset_id is None` `return None`；删 `_handle_via_legacy_llm` 物理分支 + `_execute_query`），agent 工具 `tool_registry.py:144` 本身已有 `free_sql_guard` 治理（`tool_registry.py:131-142`），本期边界内**不改 agent 工具**，只保证全局会话不进 agent 物理通道。

## Architecture Patterns

### 当前三层回退主链（`send_message_handler.py:66-84`）
```
handle()
 ├─ _handle_via_semantic_router (L1→L2 主链, :89)  ── 写死 viewer_roles=[] (:101) ← 治理缺口
 │    ├─ execute_plan(question, viewer_roles=[], runtime_mode="official") (:99)
 │    ├─ 无 execution_results → _build_blocked_route_message (:108) → blocked / None
 │    └─ 有结果 → 落 Message(source='semantic') + AgentQueryLog (:138-168)
 ├─ _handle_via_agent (:202)  ── dataset_id is None 时 channel 抛错被吞 ← 死路径
 │    └─ channel.to_agent_request → build 物理 adapter (datachat_channel.py:91) ← 物理旁路②
 └─ _handle_via_legacy_llm (:311)  ── 回退
      ├─ dataset_id is None → 诚实兜底块 (:313-332, source='legacy_llm') ← 须改 fallback
      └─ dataset_id 有值 → generate_sql + _execute_query (:355/:364) ← 物理旁路①
```

### Pattern 1: 治理对齐（方案 B，推荐）
**What:** 在 `_handle_via_semantic_router` 内，用 interfaces 透传进来的 `principal_context`/`viewer_roles` 经 `AccessPolicyDecisionService` 做 pre_route（principal 校验）与 post_compile（资源裁决），裁决 deny 时落诚实兜底而非伪造出数。
**When to use:** DataChat 主链需要与 `/agent/semantic/plan` 一致的 `policy_decision`。
**关键事实:** `execute_plan` 返回结构含 `plan`（`preview_service.py:631`），plan 内有 `execution_targets`；但 `execute_plan` **本身不调 `AccessPolicyDecisionService`**——它只在 router 内用 `policy_guard_service`（`preview_service.py:888-906`，那是 ontology 的 `PolicyGuardService`，**不同于** governance 的 `AccessPolicyDecisionService`）。这正是两条管线不一致的根因。对齐需在 handler 显式引入 `AccessPolicyDecisionService`（DI 注入）。
**编译产物来源:** post_compile 需要 `compiled_targets`。两条路：
1. 复用 `execute_plan` 已产出的执行结果/编译信息（需确认 `execution_results[i]` 是否含 `resource_set`/`sql_hash` 供 `_normalize_resource_set` 用，见 Open Question 1）；
2. 或参照 `agent_plan_handler.py:61-69` 用 `compiler_service.compile_preview` 重新编译——但这会重复编译。**规划阶段须在 GREEN plan 前确认 execute_plan 输出能否直接喂 post_compile。**

### Pattern 2: principal 透传（决策 4 瘦身）
**What:** `SendMessageCommand` 加两个可选字段 `principal_context: dict | None`、`viewer_roles: list[str] | None`；`conversations.py` 用 `principal_context_from_bearer()` 解析后填入 command；handler 透传给 `execute_plan` 和 `AccessPolicyDecisionService`，不在 handler 注入 `PrincipalResolver`。
**Anchor:** router `_resolve_roles`（`preview_service.py:823-838`）已从 `principal_context['roles']` + `viewer_roles` 取角色；`AccessPolicyDecisionService.pre_route/post_compile` 接 `PrincipalContext`，需先 `PrincipalResolver().resolve(...)` 归一——这一步可在 interfaces 层（`principal_context_from_bearer` 已返回 dict）或 handler 入口做一次 resolve。**建议在 interfaces 层 resolve 成 dict 透传，handler 内若需 `PrincipalContext` 再 `PrincipalResolver().resolve(principal_context=...)`**（纯函数不碰 g，不违反决策 4 的"不注入 resolver 到 handler 构造"——决策 4 禁的是构造注入，不是禁用纯函数）。

### Anti-Patterns to Avoid
- **改 policy `subject_roles=[]` 全放行**：破坏治理语义，等于关治理。应配主体角色绑定。
- **保留 env 灰度开关**（`DATACHAT_LEGACY_PHYSICAL_SQL_ENABLED`）：决策 2 明令禁止。
- **principal 解析下沉 application**：违反分层；碰 `g` 只能在 interfaces。
- **整体委托 AgentPlanHandler**：丢 DataChat 消息落库/trace 结构。

## Common Pitfalls

### Pitfall 1: observe 模式被误以为能救"无匹配 policy"的 deny
**What goes wrong:** 以为把 `RLS_ENFORCEMENT_MODE` 设 observe 就能放行 DataChat 出数。
**Why it happens:** observe 的"不阻断"只在 `_post_compile_with_repository` 的 **row_scope 求值阶段**生效（`access.py:749-802`：`enforcing = mode in {deny,enforce}`，observe 走 advisory 放行）。而"无匹配 allow policy → deny"（`access.py:693-714`）发生在 row_scope **之前**，是硬 deny，与 RLS 模式完全无关。
**How to avoid:** 出数的钥匙是**主体角色匹配 allow policy**（配 `data_m1_reader`），不是改 RLS 模式。规划须把角色绑定列为出数前提。
**Warning signs:** 集成测试里 principal 无 data 角色却期望 allow → 必 `data_policy_not_matched`。

### Pitfall 2: 阶段 0（治理）与阶段 1（物理收口）分开上线 → 合规错觉
**What goes wrong:** 先上 principal 注入、后删物理路，中间态"看似对齐实则未对齐"。
**Why it happens:** principal 注入单独先行，但物理旁路还在 → 治理 deny 时 legacy 仍偷偷出数（绕治理）。
**How to avoid:** CONTEXT specifics 已明令阶段 0 + 1 **同批上线**。规划须把治理对齐 + 物理删除放同一 wave 的同一原子批次。

### Pitfall 3: 集成测试 `:313` 不在改造文件清单 → make test 直接红
**What goes wrong:** `test_datachat_official_consume.py:313 assert ai["source"] == "legacy_llm"`，物理收口后该路径 source 变 `fallback`。
**How to avoid:** 该文件**必须**进改造 files；本研究 Q6 已列出全部 4 处断言改写方向。

### Pitfall 4: 全局会话进 agent 物理通道残留 running AgentQueryLog
**What goes wrong:** `_handle_via_agent`（`:202`）对 `dataset_id is None` 会让 `channel.to_agent_request` 在 `find_by_id(None)` → `if not dataset: raise`（`datachat_channel.py:72-73`）抛错；但此前已 `db.session.add(log_entry, status="running")` + commit（`:235-243`），抛错被外层 `try/except` 吞掉（`:80-81`），残留 running 日志。
**How to avoid:** 决策 5 已定：`_handle_via_agent` 开头对 `conversation.dataset_id is None` 直接 `return None`，在建 log 前短路。

## 关键研究问题逐一坐实（带 file:line）

### Q1 治理"全 deny → 答不出"风险（最高优先级）—— 已坐实

**(a) 默认 deny 的确切逻辑与触发条件：**
`_post_compile_with_repository`（`access.py:589`）流程：infer data_level（`:598`）→ 列 active policies（`:665`）→ `matching = [p for p in policies if p.matches(...)]`（`:666-675`）→ 找 deny policy（`:676`）→ **找 allow policy（`:692`）；`allow_policy is None` 时 `:693-714` 硬 deny `data_policy_not_matched`**。
触发条件：`DataPolicy.matches`（`entities.py:68-87`）在 `subject_roles` 非空且与 `principal_roles` 无交集时 `return False`（`:79`）。答题 cube → M1（`access.py:264`），唯一 M1 allow policy `m1_aggregate_read` 要求 `data_m1_reader`（`seed.py:605`）。handler 写死 `viewer_roles=[]`（`send_message_handler.py:101`）→ principal 无 data 角色 → matching 空 → deny。

**(b) 为何 observe 在 router 路径仍阻断、gateway 不阻断 —— 真相纠正：**
不是 observe 在 router 阻断、gateway 放行。真相是：**deny `data_policy_not_matched`（`:693`）与 RLS 模式无关**，发生在 row_scope 求值之前。observe 的"不阻断"只在 row_scope 阶段（`access.py:749-802`：`enforcing = mode in RLS_ENFORCING_MODES={deny,enforce}`；observe → advisory 放行）。所以"observe 仍阻断"= deny 不在 observe 作用域。**这条须在规划写清，避免被误导去调 RLS 模式。** （记忆里"observe 在 router 路径仍阻断"的表述不精确，本研究纠正为"该 deny 与 observe 无关"。）

**(c) 让已发布 cube 能被 DataChat 主体问出数的最小方案：**
答题表 M1 → 主体需持 `data_m1_reader`。现成种子已有该 allow policy（`m1_aggregate_read`，`seed.py:602-615`）+ execution_profile（`mc_m1_reader`，`seed.py:559`）+ 权限包（`catalog.py:96`，`data_m1_reader` 含 `data_m0_reader`）。**最小方案 = 给 DataChat 登录主体绑定 `data_m1_reader` 角色**（数据动作，非代码）。若答题相关还触及 M2 dwd 表，需 `data_m2_detail_reader`（`m2_detail_read` policy `seed.py:617`，row_scope `on_missing=unrestricted` 不阻断）。**不要**改 policy `subject_roles=[]`（破坏治理）。

**(d) 现成种子模板可复用：** `m0_public_read`/`m1_aggregate_read`/`m2_detail_read`/`m3_raw_block` 已在 `seed.py:585-658`，`m2_detail_read` 含 row_scope 示例。无需新建 policy，只需配主体角色绑定。

**可执行最小方案（原子批次）：** ① handler 传真实角色（透传 principal_context）；② handler 经 `AccessPolicyDecisionService` pre_route/post_compile；③ D4 给目标主体绑 `data_m1_reader`；④ 确认 observe 不影响（M1 无 row_scope，本就 allow）。
