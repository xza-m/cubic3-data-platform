## 关键归档（决策修订第 1 项）——治理裁决未在 handler 重复实现

实测 `app/application/execution_compiler/runtime_service.py:113` 已调 `AccessPolicyDecisionService.post_compile(...)`，`:126` `decision != "allow" → status='blocked'`。即 DataChat 的 `execute_plan → runtime_service.execute` 链路早已走 post_compile 治理裁决并强制 deny。本 wave **只补 principal 透传**，让真实角色到达 `PrincipalResolver.resolve`（只读 `principal_context.roles`）；**未在 handler 引入 `pre_route`/`post_compile`/`compile_preview`/`PrincipalResolver()`**（grep 防回归断言保持为 0），避免与 runtime_service 重复裁决造冗余。后续读者勿误以为「DataChat 主链零治理」而重复引入 post_compile。

## _handle_via_legacy_llm 收口（兜底落点确定性）

`_handle_via_legacy_llm` 是 `handle()` 终点（`-> Dict` 非 Optional）。删 else 物理体后两条路径均显式 return 不落空：

- `dataset_id is None`（全局会话）→ `return self._build_unanswerable_fallback(command, conversation, user_message, reason="未命中已发布语义资产")`
- `dataset` 有值会话 → `return self._build_unanswerable_fallback(command, conversation, user_message, reason="未命中已发布语义资产")`

handler 内 `return self._build_unanswerable_fallback` 计数 = 4（两处 semantic_router 分流 deny/未命中 + 两处 legacy 终点），满足 >= 2。

## _build_unanswerable_fallback 文案与三处收敛点

文案：「未能在已发布的语义资产中找到可回答该问题的口径。请换种问法，或确认相关 Cube / 指标已发布到语义中心。」（`source='fallback'`、`error=reason`、`_record_query_log(status='unanswerable')`）

三类答不出收敛点：
1. **治理 deny**：`execution_results` 非空但 `primary_result.status=='blocked'`（下游 post_compile deny）→ fallback（reason=primary_result.reason，如 `data_policy_not_matched`）。
2. **纯未命中**：`execution_results` 空且 `_build_blocked_route_message` 返 None（reason 以「未命中」开头）→ fallback。
3. **主链全失败**：semantic/agent 均未作答，落 `_handle_via_legacy_llm` 终点（None / 有值两路）→ fallback。

> 保留区分：route 级「运行时未就绪」（reason=semantic_runtime_not_ready）仍由 `_build_blocked_route_message` 诚实回 `source='semantic'`（Test B 不动），不并入 fallback。
