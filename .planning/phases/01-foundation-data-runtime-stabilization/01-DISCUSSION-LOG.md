# Phase 1: 基础接入与运行底座稳定化 - Discussion Log

**Date:** 2026-03-25
**Status:** Completed

## Summary

本次讨论聚焦 Phase 1 的实现边界，不新增能力范围，只锁定数据中心主链路应如何稳定落地。所有选择都服务于后续 `gsd-plan-phase 1` 的任务拆解。

## Decisions by Area

### 1. 数据源验收基线
- 讨论项：Phase 1 应以哪些异构数据源作为稳定可用的验收基线。
- 结论：选择 `PostgreSQL + MaxCompute`。
- 理由：分别代表最常见的 `OLTP` 与 `OLAP` 场景，也是当前最常用的数据库和数仓类型。

### 2. 元数据同步策略
- 讨论项：只做手动同步，还是加入自动首次同步与自动定时同步。
- 结论：采用“自动首次同步 + 手动重同步 + 平台统一固定周期自动同步”。
- 补充：不做每个数据集单独配置调度。

### 3. 自动同步覆盖范围
- 讨论项：自动同步只刷元数据，还是同时覆盖注册对象。
- 结论：自动同步同时覆盖数据源目录刷新和已注册数据集同步。

### 4. 预览深度
- 讨论项：只展示字段列表，还是补齐更完整的预览与失败反馈。
- 结论：预览必须包含表级信息、字段识别、分类型失败反馈，以及 `LIMIT 20` 样本数据。

### 5. 部署口径
- 讨论项：Phase 1 是否把部署体验作为关键验收项。
- 结论：不把部署体验作为重点目标。
- 补充：后续只要求现有容器体系能支撑联调与功能验证，不追求一键部署。

### 6. 数据集类型范围
- 讨论项：只打穿物理表，还是同时要求 SQL 与文件数据集稳定可用。
- 结论：三种数据集类型都要稳定可用：物理表、SQL 虚拟数据集、文件数据集。

### 7. 长耗时交互模式
- 讨论项：同步等待、全部异步、还是混合模式。
- 结论：采用混合模式。
- 细化：
  - 连接测试、轻量预览：同步返回。
  - 首次同步、手动重同步、定时同步：后台任务 + 状态轮询 / 回写。

### 8. 状态与失败信息展示层级
- 讨论项：只在详情页展示完整错误，还是全页面铺满细节。
- 结论：列表页展示摘要，详情页与注册页展示完整失败原因与最近执行结果。

### 9. SQL 虚拟数据集定位
- 讨论项：轻量注册对象、受管对象、或完整治理对象。
- 结论：按受管数据集对象处理。
- 包含：保存 SQL、字段解析、样本预览、编辑后重解析、手动刷新、清晰状态与失败原因。
- 不包含：复杂 lineage、依赖图、版本治理。

### 10. 文件数据集边界
- 讨论项：支持哪些格式，以及是否允许覆盖原数据集。
- 结论：支持 `CSV + Excel`，且不允许覆盖原数据集；重新上传必须创建新数据集。

### 11. 三类数据集的一致性策略
- 讨论项：完全统一体验、统一骨架，还是各做各的。
- 结论：采用“统一骨架、类型差异可见”。
- 统一内容：列表状态模型、详情基础信息区、失败表达方式、通用操作入口位置。
- 允许差异：注册流程、特有字段、特有操作。

### 12. 自动定时同步失败后的处理
- 讨论项：失败即人工处理，还是带重试与暂停策略。
- 结论：自动重试 1 次；仍失败则标记失败并等待人工处理。

## Deferred Ideas

- SQL 虚拟数据集更完整的治理能力。
- 文件数据集覆盖更新。
- 更多文件格式支持。
