# 研究：常见坑点

**领域：** 语义中心数据平台、内部分析产品、AI 辅助数据工作流  
**研究时间：** 2026-03-25  
**置信度：** HIGH

## 关键坑点

### 坑点 1：语义口径分叉，指标“看起来一致”但实际不一致

**会发生什么：**  
同一个业务指标在不同页面、不同 API、不同 AI 问答里被重新实现，导致“订单数、活跃数、留存率”在不同入口上不一致。短期看像是查询结果差异，长期会演化为业务对平台失去信任，最终回到手写 SQL 和 Excel。

**为什么会发生：**  
语义层只做了“字段收口”，没有把指标定义、过滤条件、时间粒度、维度约束、口径版本一起收口；或者团队允许临时 SQL、临时派生指标、临时小组件绕过主语义层。内部分析产品越快上线，越容易出现多个“事实来源”并存。

**如何避免：**  
把语义层当作唯一口径入口，强制指标定义版本化、可追溯、可比较。新指标先进入语义仓库，再暴露给查询中心、应用模板和 AI 工具；禁止业务层直接复制口径。对核心指标建立“定义即测试”，至少覆盖粒度、过滤、去重、关联链路和历史回放。

**早期预警信号：**  
同一指标在 2 个页面以上出现轻微差异；分析师开始口头解释“这个页面口径不同”；需要频繁补充手工说明；同一个 AI 问答问题在不同时间返回不同答案。

**应对阶段：**  
Phase 1.1 / 1.2（语义模型和字段校验）、Phase 1.6（口径回归测试）、应用中心 Phase 1（应用定义与执行器入库前校验）

---

### 坑点 2：Join 图和数据粒度没建完整，查询信任会被 fan-out 直接打穿

**会发生什么：**  
查询看起来“能跑”，但一旦跨事实表、枚举维表、层级维表、宽表混合，结果就会重复计数、金额膨胀、分组错位。用户通常不会马上发现，直到和财务、运营、风控对账时才暴露。

**为什么会发生：**  
团队只关注“能连上”，没有把主键、基数关系、可达路径、默认粒度、子查询重写和歧义消解定义清楚。尤其在 AI 生成查询时，模型会倾向于选择“看起来最合理”的 join，而不是“语义上唯一正确”的 join。

**如何避免：**  
为每个 Cube/View 定义明确 grain、主键、relationship、默认聚合策略和受限 join 路径；编译器必须把 fan-out 风险当作硬错误或强制子查询重写，而不是静默放行。对核心链路做“对账型”测试集，重点覆盖一对多、多对多、空维度和时间粒度切换。

**早期预警信号：**  
某些查询结果比历史值系统性偏大；同一用户在不同维度下总数对不上；AI 问答回答频繁出现“可能是因为 join”；编译器开始积累“暂时放行”的白名单。

**应对阶段：**  
Phase 1.3（JoinGraph + Compiler + fan-out 防护）、Phase 1.6（端到端对账测试）、后续任何新增 Cube/View 的设计评审

---

### 坑点 3：物理层同步只做“定时检查”，没做“发布门禁”

**会发生什么：**  
语义定义已经改了，但底层表结构、字典表、分区字段、枚举值还没跟上；或者物理层已变更，但语义层、缓存层、文档和示例还沿用旧版本。结果是“线上报错、离线看着正常”，排查成本很高。

**为什么会发生：**  
团队把同步当成运维任务，而不是发布流程的一部分。只做每日巡检而不做发布前校验，就会把漂移从“可控延迟”变成“上线事故”。

**如何避免：**  
把 schema diff、字段引用校验、字典映射校验、物理表存在性检查放进发布链路；定时任务只负责补充发现，不负责兜底。关键字段变更要有双轨期：先兼容、再切流、最后清理旧版本。

**早期预警信号：**  
同步任务总是发现问题但无人处理；“临时修一下表结构”出现频率上升；语义定义与仓库表结构的变更提交不在同一批次；旧字段被连续多个新任务复用。

**应对阶段：**  
Phase 1.5（物理层同步）、Phase 1.6（变更回归）、任何仓库 schema 变更阶段

---

### 坑点 4：内部应用市场先长出“模板森林”，后补治理

**会发生什么：**  
相似应用被复制成多个模板分支，配置 schema、默认参数、执行器行为各不相同。用户以为自己在复用标准应用，实际上是在使用互不兼容的派生版本。后续每次改动都要同步多个模板，维护成本指数级上升。

**为什么会发生：**  
内部平台很容易把“快速出 Demo”误当成“平台化能力”。如果模板没有所有权、版本、兼容策略和弃用流程，就会从“可配置产品”退化为“复制粘贴仓库”。

**如何避免：**  
把模板分成“核心模板、受控扩展、实验模板”三类；核心模板必须可追踪版本，扩展点必须显式声明。新增模板前先问是否能通过参数化、组件化、执行器复用解决，而不是再复制一份。

**早期预警信号：**  
相似应用数量快速增长；同类模板的配置 schema 差异越来越大；文档里开始出现“这个模板是那个模板的变体”；修一个 bug 要改多处定义。

**应对阶段：**  
应用中心 Phase 1-2（应用定义和前端模板进入治理）、Phase 4（内置应用收口）、Phase 5（监控与管理）

---

### 坑点 5：Demo 逻辑直接上线，预览数据和生产数据混为一谈

**会发生什么：**  
演示时运行顺畅，但一旦接入真实权限、真实数据量、真实刷新周期和真实失败场景，体验就崩掉。最常见的是：Demo 用静态样例，生产依赖异步刷新；Demo 省略权限校验，生产却要求细粒度授权；Demo 能接受几秒延迟，生产用户却默认即时可见。

**为什么会发生：**  
团队在前期为了速度，会默认“先做成能演示的样子”。如果没有明确的 demo/prod 分层，后来很容易把占位实现、预置数据和预览模式当作正式能力。

**如何避免：**  
把预览、沙箱、生产三种模式明确分开；每个模式都要有独立的数据源、权限和错误提示。任何 preview/beta 能力都不能默认承诺生产可用，必须有上线条件、SLA 和回滚策略。

**早期预警信号：**  
演示数据总是手工维护；用户开始要求“把 demo 直接开到生产”；刷新失败时只能靠人工补数据；测试环境和生产环境的配置差异越来越大。

**应对阶段：**  
Phase 2-4（前端模板、执行引擎、内置应用）、任何新功能从 Demo 到生产的切换窗口

---

### 坑点 6：AI 辅助查询没有“可验证输出”，只剩下“会说话的假答案”

**会发生什么：**  
模型能生成看起来很合理的 DSL、SQL、摘要或解释，但真实语义可能有偏差，尤其在工具选择、字段映射、过滤条件和时间窗口上。用户第一次会觉得“挺聪明”，第二次开始发现“不稳定”，第三次就不敢再用。

**为什么会发生：**  
把 LLM 当成查询引擎而不是受约束的规划器。没有严格输出格式、没有工具调用边界、没有结果校验和失败回退时，模型会把“补全语句”当成“理解业务”。

**如何避免：**  
对 AI 输出使用结构化约束和严格校验，只允许它生成受控 DSL 或受控参数，再交给编译器与执行器；把查询结果、解释文本和置信信息拆开处理。高风险操作必须有人工确认或二次校验，不能直接自动执行。

**早期预警信号：**  
模型经常补全出不存在的字段；同一问题在不同轮次回答不同；工具调用失败后模型仍尝试编造结果；业务用户开始拿 AI 结果做正式决策前要自行二次验证。

**应对阶段：**  
Phase 1.4（Agent 工具集成与系统提示）、Phase 1.6（有/无 Recipe 对比和准确率评估）、后续所有 DataAgent 场景

## 技术债模式

短期看起来合理，但会把平台做成“可跑不可控”。

| 取巧方式 | 立刻收益 | 长期代价 | 何时可接受 |
|---|---|---|---|
| 允许业务层直接写临时 SQL 绕过语义层 | 上线快、修 bug 快 | 口径分叉、审计困难、后续难收口 | 仅限一次性排障，且必须补回语义定义 |
| 复制模板再改配置而不是复用执行器/Schema | Demo 快、验证快 | 模板分裂、回归成本高 | 仅限受控实验模板，且有淘汰日期 |
| AI 查询失败后自动降级为自由文本答案 | 用户“感觉有答案” | 可信度崩坏、误导决策 | 几乎不接受；仅限明确标注的探索性问答 |
| 预览模式直接共用生产数据源 | 少做一套环境 | 权限、刷新、性能和审计问题集中爆发 | 仅限离线只读预览，且不可混用写操作 |

## 集成坑点

| 集成对象 | 常见错误 | 正确做法 |
|---|---|---|
| 语义仓库 / Cube 定义 | 只同步字段，不同步粒度、主键、关系 | 把模型、关系、口径、测试一起同步 |
| 查询编译器 / 执行器 | 把 join 选择交给“默认最短路径” | 对歧义路径强制显式规则或报错 |
| AI Agent / 工具调用 | 直接让模型输出可执行 SQL | 先输出受控 DSL/结构化参数，再编译执行 |
| 应用市场 / 模板系统 | 把每个应用都做成独立副本 | 优先参数化、组件化、版本化模板 |

## 性能陷阱

| 陷阱 | 症状 | 预防 | 何时开始失控 |
|---|---|---|---|
| Join 图爆炸 | 复杂查询延迟明显上升 | 限制可达路径、缓存编译结果、对子查询分层 | 当 Cube / 维表数量增加且路径歧义开始出现时 |
| 语义缓存失效策略过宽或过窄 | 结果要么陈旧，要么频繁回源 | 以语义版本、数据版本和参数签名联合失效 | 当同一指标存在多版本发布时 |
| 模板渲染链路过重 | 打开应用详情页很慢 | 组件化配置与懒加载 | 当模板数量和配置项明显增长时 |
| AI 问答无结果压缩 | 上下文过长、成本高、响应慢 | 只传必要语义摘要和结构化结果 | 当问答开始带多轮上下文和大结果集时 |

## 安全错误

| 错误 | 风险 | 预防 |
|---|---|---|
| 让 AI 直接读取未授权数据源 | 语义泄露、越权访问 | 工具层做权限裁剪，模型永远拿不到原始越权连接信息 |
| 在语义层或模板中嵌入可执行任意 SQL | 注入、跨租户数据暴露 | 只允许受控 DSL，所有自由文本必须经过白名单解析 |
| 共享 demo 账户或静态 token | 审计失真、权限失控 | 每个环境独立凭证，演示数据和生产数据隔离 |
| 将 prompt 当成配置文件直接拼接 | prompt injection、工具滥用 | 对外部内容做转义与上下文隔离，敏感工具二次确认 |

## 体验坑点

| 坑点 | 用户影响 | 更好做法 |
|---|---|---|
| 查询结果只给数值，不给口径和新鲜度 | 用户无法判断能不能信 | 同步展示口径版本、更新时间、粒度和限制 |
| AI 回答不给可追溯依据 | 用户不敢拿去决策 | 结果旁边显示命中的语义对象和执行路径 |
| 模板库卡片很多，但不知道该用哪个 | 选择成本高，流失到外部工具 | 提供按场景、角色、数据域筛选 |
| 出错时只显示“执行失败” | 用户不知道该改什么 | 错误分成口径、权限、数据、编译、执行五类 |

## “看起来完成了但其实没有”检查清单

- [ ] **语义层已上线：** 常常缺少 grain、主键、关系测试，先验证 fan-out 和口径一致性。
- [ ] **应用市场可用了：** 常常缺少模板版本、权限边界和弃用流程，先验证是否能安全复用。
- [ ] **AI 问答能回答：** 常常缺少结构化输出校验、工具失败回退和结果可追溯性，先验证输出是否可执行且可解释。
- [ ] **同步任务在跑：** 常常缺少发布门禁和回滚策略，先验证 schema 漂移是否会阻断上线。

## 恢复策略

| 坑点 | 恢复成本 | 恢复步骤 |
|---|---|---|
| 语义口径分叉 | HIGH | 锁定权威口径，冻结新增派生指标，补测试，逐步迁移引用方 |
| Fan-out 导致错数 | HIGH | 回溯 join 路径，修正 grain 和 relationship，补对账样例，禁止静默放行 |
| 物理层与语义层漂移 | MEDIUM-HIGH | 先做兼容字段，再做同步修复，最后清理旧字段和旧模板 |
| 模板森林失控 | MEDIUM | 收口到少量核心模板，废弃重复模板，统一执行器和配置 Schema |
| AI 假答案进入决策链 | HIGH | 降级为只读/建议模式，重新引入结构化约束和人工复核 |

## 坑点到阶段映射

| 坑点 | 预防阶段 | 验证方式 |
|---|---|---|
| 语义口径分叉 | Phase 1.1 / 1.2 | 指标定义、YAML Schema、口径回归测试通过 |
| Join 图和粒度错误 | Phase 1.3 | fan-out 测试、歧义 join 测试、对账样例通过 |
| 物理层同步不足 | Phase 1.5 | schema diff 和漂移告警在发布前阻断问题 |
| 模板森林 | 应用中心 Phase 1-2 / 4 | 模板版本、复用率、弃用流程都可追踪 |
| Demo 与生产错位 | Phase 2-5 | 预览/沙箱/生产三态隔离验证通过 |
| AI 查询不可信 | Phase 1.4 / 1.6 | 结构化输出、工具调用、准确率和回归评估通过 |
