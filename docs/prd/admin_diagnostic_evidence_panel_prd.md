# 管理员诊断证据面板设计

## 背景

平台已经完成一轮 UI 治理：普通业务路径不再直接展示 API 路由、底层 ID、内部枚举、Gateway Query ID、SQL / YAML / JSON 等实现细节。但这些信息并不能简单删除。它们在管理员排障、语义建模审阅、审计追溯、查询诊断和网关联调中仍有业务价值。

本设计定义一套统一的“诊断证据面板”模式，把技术证据从普通业务界面中收敛到有权限、有语境、有脱敏、有审计的高级诊断入口。

## 目标

- 让普通用户只看到业务对象、状态、风险和下一步动作。
- 让管理员、数据工程师和平台工程师能在同一套模式下查看 SQL、YAML、JSON、Trace、Gateway telemetry、API Key 引用和执行审计。
- 避免各页面继续零散铺开代码块、原始 payload、底层枚举和日志。
- 建立统一的脱敏、复制、下载、展开、审计提示和权限边界。
- 保持现有功能模块边界：诊断入口跟随业务对象出现，不把数据中心、语义中心、应用中心、配置中心和查询中心强行合并成一个跨域工作台。

## 非目标

- 不新增一个取代各模块的“统一诊断工作台”。首期只做模块内嵌的统一证据模式。
- 不隐藏查询工作台中的 SQL。SQL 是查询产品的核心对象，应保留在主路径，但需要更清晰的标签、格式和风险提示。
- 不把前端隐藏作为安全措施。所有敏感信息必须由后端按角色、权限和脱敏策略返回。
- 不新增第二套审计事实源。诊断面板只展示已有事实：语义 Registry、Gateway telemetry、governance audit、运行记录、配置引用等。
- 不在本阶段重做全局设计系统，只定义可复用模式和组件契约，落地时复用现有组件。

## 产品形态定位

本设计不是新增一个跨模块工作台，而是定义“分布式入口 + 统一证据模式”。

### 保持分布式入口

诊断入口跟随用户当前正在处理的对象：

- 数据中心的数据源、数据集、同步任务，在数据中心内查看诊断证据。
- 语义中心的 Cube、Ontology、建模 Copilot、DevTools，在语义中心内查看诊断证据。
- 应用中心的应用实例、执行记录、订阅投递，在应用中心内查看诊断证据。
- 配置中心的权限、网关观测、渠道、订阅，在配置中心内查看诊断证据。
- 查询中心的 SQL 执行、导出、保存查询，在查询中心内查看执行和治理证据。

这样做的原因是用户心智以业务对象为中心：用户想知道“这个 Cube 为什么发布失败”“这个数据源为什么连不上”“这次执行为什么没推送”，而不是先进入一个新的总工作台再反查对象。

### 统一的是证据模式

统一部分只包括：

- 证据类型：summary、lineage、policy、execution、code、trace、secret_ref。
- 展示组件：诊断抽屉、代码证据块、Trace 时间线、密钥引用卡。
- 权限策略：business / diagnostic / raw 三层视图。
- 脱敏与复制规则：所有原始证据默认折叠，复制内容必须是脱敏版本。
- 测试契约：普通视图不外露 raw 证据，有权限时能打开诊断面板。

### 可选的后续聚合入口

后续如果确实需要跨模块查询，可以只在配置中心或运维区增加一个“诊断记录检索”入口，用于按 trace、对象、用户、时间检索已有证据。它是索引和检索能力，不是替代各模块的主工作台，也不承载日常业务操作。

## 用户与权限分层

| 角色 | 默认可见内容 | 可申请/可展开内容 | 禁止直接展示 |
| --- | --- | --- | --- |
| 业务用户 | 业务摘要、状态、影响范围、风险等级、下一步动作 | 无 | SQL、YAML、JSON、Trace、密钥、内部 ID |
| 数据建模工程师 | 语义资产摘要、字段候选、绑定结果、校验问题 | 语义 Spec、字段证据、发布预演 Trace | 密钥明文、跨域无关执行日志 |
| 数据平台管理员 | 运行状态、审计摘要、策略命中、渠道/订阅诊断 | SQL、Trace、Gateway telemetry、原始 payload 脱敏视图 | 密钥明文 |
| 平台工程师 | 全量诊断摘要、服务健康、契约状态 | 脱敏原始 payload、请求/响应 Trace、配置引用 | 密钥明文 |

权限原则：

- `business` 视图默认给所有登录用户。
- `diagnostic` 视图只给具备平台管理、语义治理或对应资源维护权限的用户。
- `raw` 视图只给平台管理员或工程诊断权限，且必须脱敏。
- `secret` 永远只显示引用名、状态和最近校验时间，不显示明文。

## 设计方案比较

### 方案 A：每个页面各自加“高级信息”折叠区

优点：

- 改动最小，单页可以快速上线。
- 不需要先抽象通用组件。

缺点：

- 容易回到各页面文案、样式、权限判断不一致的问题。
- SQL / YAML / JSON / Trace 的复制、脱敏、空态、错误态会重复实现。
- 后续很难统一审计口径。

适用场景：

- 单个高风险页面临时止血。

### 方案 B：模块内嵌“诊断证据面板”组件与统一证据模型

优点：

- 普通视图和诊断视图边界清晰，用户心智稳定。
- 证据展示、脱敏、复制、下载、审计提示和权限控制可以复用。
- 后续新增页面只需要提供证据模型，不再重新设计原始信息展示。
- 不改变数据中心、语义中心、应用中心、配置中心和查询中心的职责边界。

缺点：

- 首期需要整理通用证据类型和页面接入清单。
- 部分页面需要把原先散落的代码块改造成结构化证据输入。

推荐：

- 采用方案 B。它更符合 SOLID 和 DRY，也能避免本轮刚清理完的问题反复出现，同时不会制造新的跨模块工作台。方案 A 只作为紧急止血方式，不作为长期平台规范。

## 信息架构

每个页面分成三层信息：

1. **业务摘要层**
   - 面向所有用户。
   - 展示名称、状态、风险、影响、最近更新、可执行动作。
   - 禁止出现 API path、底层枚举、数据库密钥、内部 trace id、完整 SQL/YAML/JSON。

2. **诊断证据层**
   - 通过“诊断证据”按钮或右侧抽屉进入。
   - 面向管理员、建模工程师、平台工程师。
   - 展示结构化证据：编译结果、策略命中、字段来源、发布预演、执行链路、gateway 遥测摘要。

3. **原始证据层**
   - 诊断面板里的二级展开。
   - 展示脱敏后的 SQL、YAML、JSON、Trace、日志片段。
   - 需要明确来源、生成时间、脱敏状态、复制/下载行为。

## 证据类型模型

统一证据类型建议如下：

| 类型 | 中文名称 | 典型内容 | 默认展示方式 |
| --- | --- | --- | --- |
| `summary` | 诊断摘要 | 状态、风险、命中策略、耗时、错误分类 | Key-value / 状态卡 |
| `lineage` | 来源证据 | 表、字段、资产引用、候选来源 | 表格 / 关系列表 |
| `policy` | 治理证据 | PolicyDecision、row scope、角色、阻断原因 | 时间线 / 命中规则 |
| `execution` | 执行证据 | Query Plan、Gateway 状态、运行耗时、结果对象 | 时间线 / 指标卡 |
| `code` | 代码证据 | SQL、YAML、JSON、DSL、请求 payload | 代码块，默认折叠 |
| `trace` | Trace 证据 | trace id、span、请求链路、错误栈摘要 | 时间线，原始内容折叠 |
| `secret_ref` | 密钥引用 | API Key 引用、CredentialBinding、Secret 名称 | 引用卡，永不显示明文 |

建议前端输入契约：

```ts
type DiagnosticEvidence = {
  id: string
  type: 'summary' | 'lineage' | 'policy' | 'execution' | 'code' | 'trace' | 'secret_ref'
  title: string
  description?: string
  severity?: 'info' | 'warning' | 'error'
  source: string
  generatedAt?: string
  redacted?: boolean
  permission?: 'business' | 'diagnostic' | 'raw'
  content: unknown
}
```

## 组件模式

### 1. `DiagnosticEvidenceButton`

- 放在页面右上角、详情页操作区或二级 Tab 末尾。
- 文案使用“诊断证据”，不使用“Debug”“Raw”“API”。
- 没权限时不展示或展示为禁用并说明“需要管理员权限”。

### 2. `DiagnosticEvidenceDrawer`

- 右侧抽屉，宽度建议 420-560px。
- 顶部展示业务对象名称、当前状态、证据更新时间。
- 第一屏只展示摘要和高价值证据，不直接铺开代码块。
- 支持证据类型 Tab：摘要、来源、治理、执行、原始证据。

### 3. `CodeEvidenceBlock`

- 支持 SQL / YAML / JSON / DSL。
- 默认折叠，显示语言、行数、是否脱敏、来源和生成时间。
- 展开后使用固定高度滚动，不撑开页面主布局。
- 复制按钮需要复制脱敏后的内容，并给出“已复制脱敏内容”提示。
- 禁止在普通卡片、表格行、空态说明中直接使用大代码块。

### 4. `TraceTimeline`

- 展示关键阶段：接收请求、身份解析、语义路由、编译、策略判断、gateway 提交、结果返回。
- 每个节点显示状态、耗时、错误摘要和证据入口。
- 原始 span / payload 只在二级展开中展示。

### 5. `SecretReferenceCard`

- 展示密钥引用名、用途、状态、最近校验时间、轮换建议。
- 永不展示 secret value。
- 复制动作只能复制引用名，不复制明文。

## 页面接入清单

### 第一阶段：高风险外露收敛

- 语义 DevTools：定位为管理员诊断工作台，保留 Trace / Preview，但统一接入诊断证据组件。
- Cube 编辑器：YAML / Spec 移入“原始证据”，主表单只保留业务字段、指标、维度和发布校验。
- 建模 Copilot：Artifact 面板保留 Review / Source / Preview / Trace，但 Trace 和 raw spec 使用统一证据面板样式。
- 数据源详情：连接配置、测试结果、Schema 浏览日志、同步错误进入诊断证据；普通视图只展示连接健康、来源类型、最近同步。
- 数据集详情：SQL / file metadata / profile 原始内容进入诊断证据；普通视图展示字段、质量、血缘和影响。

### 第二阶段：运行与审计证据统一

- 抽取任务详情：运行日志、原始 payload、错误栈进入诊断证据；主视图展示运行状态、输入输出、失败原因和重试动作。
- Gateway 观测：Trace 查询作为明确管理员 Tab；具体 trace 详情进入 `TraceTimeline`。
- 权限管理：行级范围摘要继续保持表格紧凑，完整策略和 row scope 进入治理证据。
- 应用执行详情：执行 payload、订阅投递日志、Webhook 响应进入诊断证据。

### 第三阶段：查询类页面信息密度治理

- 查询工作台：SQL 是主对象，保留主编辑器；执行 trace、治理证据、gateway 结果对象进入诊断证据。
- 保存查询 / 导出任务：保留查询名称、用途、调度、结果；SQL 默认折叠，按“查看查询语句”进入代码证据。

## 文案规则

- 普通视图避免出现：`API`、`route_type`、`payload`、`trace_id`、`raw_spec`、`app_code`、`gateway query id`、`mc_m*_reader`。
- 管理员视图可以出现：`Trace`、`Gateway`、`SQL`、`YAML`、`JSON`、`API Key`，但必须有上下文说明。
- `Schema` 在数据源、语义建模、应用配置结构中可保留，但普通用户路径优先显示“结构”“字段结构”“配置结构”。
- 错误提示优先给行动建议：例如“字段缺失，重新同步元数据”优于“view_binding_invalid”。
- 代码块标题使用“生成 SQL”“语义 Spec”“请求参数”“响应摘要”，不使用“Raw JSON”作为首屏标题。

## 安全与审计要求

- 所有密钥、token、cookie、授权头、连接密码由后端脱敏后返回。
- 前端只负责二次保护：隐藏、折叠、复制脱敏内容、禁止误展示。
- 访问 `raw` 证据时，后端应记录审计事件：访问人、资源、证据类型、时间、是否下载。
- 下载原始证据必须二次确认，默认文件名不包含敏感业务词和用户标识。
- 无权限用户不应通过 URL、前端状态或 mock 字段看到原始证据。

## 空态与异常态

- 无证据：显示“当前对象暂无诊断证据”，并给出可触发的动作，如“重新运行校验”“刷新遥测”“同步元数据”。
- 无权限：显示“需要管理员或资源维护权限查看诊断证据”。
- 证据过期：显示生成时间和刷新入口，不直接用旧证据指导用户操作。
- 脱敏失败：不展示原文，只显示“证据包含敏感字段，已阻断展示”。

## 验收标准

- 普通业务页面首屏不出现内部路由、技术 ID、底层枚举和大段 SQL/YAML/JSON。
- 管理员能在统一入口查看必要证据，并知道证据来源、时间和脱敏状态。
- 代码证据不会撑高表格行或主内容布局。
- API Key、Credential、Token 永不明文展示。
- 关键页面有回归测试覆盖：无权限不展示 raw 证据，有权限可打开诊断面板，复制内容为脱敏版本。
- e2e smoke 覆盖至少一条语义资产、一条数据源、一条 gateway trace、一条应用执行诊断路径。

## 分阶段落地计划

### P0：设计固化与组件骨架

- 固化本 PRD。
- 新增 `DiagnosticEvidenceDrawer`、`CodeEvidenceBlock`、`TraceTimeline`、`SecretReferenceCard` 的组件骨架和 Story / 单测。
- 先用 fixture 证据模型驱动，不改后端契约。

### P1：语义与数据资产接入

- 接入 DevTools、Cube 编辑器、建模 Copilot、数据源详情、数据集详情。
- 清理这些页面里散落的原始 SQL/YAML/JSON/Trace 展示。
- 补单元测试和页面 smoke。

### P2：运行与治理接入

- 接入 Gateway 观测、权限管理、抽取任务、应用执行详情。
- 补充后端证据脱敏和访问审计。
- 将 row scope、execution profile、gateway telemetry 统一转成证据模型。

### P3：查询类页面密度治理

- 查询工作台保留 SQL 主编辑器，但将执行证据、治理证据和 gateway 结果对象统一收敛。
- 保存查询和导出任务默认折叠 SQL，减少列表和详情信息噪声。

## 工程原则检查

- KISS：用三层信息模型解决“业务视图”和“诊断证据”混杂问题，避免每页重新解释。
- YAGNI：首期不设计复杂权限后台和证据仓库，只消费已有事实源并做展示治理。
- SOLID：证据渲染组件只负责展示，权限和脱敏由后端/应用服务提供，页面只组装业务对象和证据入口。
- DRY：SQL/YAML/JSON/Trace/Secret 引用统一走组件和证据模型，避免各页面重复写代码块、脱敏提示和复制逻辑。
