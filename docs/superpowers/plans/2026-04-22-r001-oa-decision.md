## 1. 背景（3 行）

- Round 3 W4 cutover 时，`codex-ontology-workbench-object-aggregate` 工作树在并行开发本体对象的「对象聚合 + 绑定健康 + 辅助索引」，

- C+ 策略：只打 `archive/ontology-object-aggregate-2026-04-14` tag 留存历史，不并 main；留给本轮做成本—收益分析。

### 2.2 API 面（增量）

- `GET /api/v1/ontology/objects/<name>/aggregate` —— **主角**：单次返回对象 + 属性 + 指标 + 关系 + 动作 + 策略 + 历史 + 审计追踪。
- `GET /api/v1/ontology/indexes/relations|rules|metrics` —— 辅助索引读模型，前端"关系/规则/指标"栏目以此为数据源，减少 N+1。
- `binding_status_helper` —— 把绑定态"是否过时"、"最新 trace" 语义提升到一处，服务于 UI 中的 *绑定修复抽屉*。

### 3.2 风险细化

- 🔴 `workbench_read_service` vs `object_aggregate_service` 语义重叠：**必须在移植前写一段"边界 ADR"**（≤1 页），否则两套聚合读模型会导致前端 / 后端维护端双倍心智。
- 🟡 `binding_health` 的"会话修复" 逻辑与 W4 之后 main 的 `policy_guard_service` 有一定重叠，需要验证 `binding_status_helper.is_stale_decision` 的 signature 是否已被 main 版本覆盖。
- 🟢 YAML seed 冲突：现有 `.gitignore` 对 `app/infrastructure/ontology/objects/pw_*.yml` / `pg_*.yml` 做了忽略，`customer.yml` / `order.yml` 是**显式入库**的业务 cube，合并时要改名或放在 `examples/` 子目录。
- 🟢 路由命名空间：`/api/v1/ontology/*` 已由 `create_ontology_blueprint` 动态装配，新增端点只需给 `create_ontology_blueprint()` 增加 `object_aggregate_service` / `binding_health_service` / `object_index_service` 三个注入点。
