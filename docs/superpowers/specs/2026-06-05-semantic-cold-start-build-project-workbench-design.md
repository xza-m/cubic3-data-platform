# 语义冷启动 Build Project 工作台产品设计

> 日期：2026-06-05
> 主题：将语义冷启动从 Copilot / 单页工作台 / 批量流水线的摇摆状态，收敛为长期 Build Project 工作台产品形态

## 1. 背景

语义层冷启动在历史实现中多次在“工作台”“Copilot”“批量冷启动”“本体工作台”“Cube 辅助建模”等形态之间切换。表面看是页面和交互变化，根因是产品主语没有稳定：

- 如果主语是 `Copilot`，用户会把 AI 生成结果误认为语义真值，字段审阅、Spec Diff 和发布门禁会被后置。
- 如果主语是单资产 `Workbench`，产品看起来规范，但不自然支撑批量冷启动。
- 如果主语是批量 `Pipeline`，规模化能力强，但单资产建模会变成流程审批，编辑手感过硬。

本设计将主产品形态收敛为：

> 语义建设工作台是长期 `Build Project` 容器；批量是项目内模式，AI 是项目内能力，`Asset Package` 内同时建设 `Cube Layer + Ontology Layer + Binding`，最终发布到语义中心。

## 2. 产品判断

### 2.1 推荐形态

采用 **A+：项目式工作台 + 批量项目模式 + 内嵌 AI 能力**。

- `Build Project` 是长期过程容器，不是语义中心真值。
- `Asset Package` 是批量和单资产之间的桥。
- `Copilot / AI` 是加速器，不作为产品主入口或发布主体。
- `Release` 是语义中心正式版本，下游消费者只消费语义中心资产。

### 2.2 为什么不是纯 Copilot

Copilot 上手轻、Demo 感强，但不适合作为规范建模产品的主心智。原因是：

- 它容易让 AI 输出变成事实源，冲击 `Spec-first`。
- 它不擅长表达几十个候选资产的状态、风险、优先级和审阅进度。
- 它会让字段候选、口径确认、编译预演和发布门禁成为附属能力。

Copilot 应保留为 AI 能力，而不是主产品对象。

### 2.3 为什么不是纯流水线

流水线适合大规模批量冷启动和团队状态追踪，但不适合作为唯一主形态。原因是：

- 单资产建模会过度流程化，用户像在跑任务而不是做建模判断。
- 前期配置门槛高，不利于探索式建设。
- 用户可能误以为系统会自动发布语义，忽略人工口径确认。

流水线应成为 `Build Project` 内的批量候选生成和状态流转模式。

## 3. 设计原则

- `KISS`：用户主线固定为“创建或继续语义建设项目 -> 生成候选队列 -> 审阅资产包 -> 发布到语义中心”，避免多个入口并列。
- `YAGNI`：冷启动阶段只做必要轻本体锚定，不把完整本体治理、复杂审批和跨域术语生命周期塞进工作台。
- `SOLID`：项目容器、资产包、字段候选、AI 建议、发布预演、语义中心 Release 各自承担单一职责。
- `DRY`：快速模式、批量模式、修订回流复用同一套 `Build Project / Asset Package / Proposal / Release` 对象模型。

## 4. 核心对象模型

```text
Build Project
  -> Asset Package[]
      -> Field Candidate[]
      -> Cube Layer
      -> Ontology Layer
      -> Binding
  -> Proposal
  -> Release
  -> Semantic Center
```

### 4.1 Build Project

`Build Project` 是长期保留的语义建设过程对象。

它负责：

- 记录建设范围：业务域、源表范围、建设策略、目标。
- 组织候选队列：事实主题、维度主题、指标组、对象绑定组。
- 保存过程状态：待审阅、需补范围、高风险、可发布、已发布。
- 承载 AI 建议：范围补全、候选解释、字段建议、风险说明。
- 串联发布前检查：Spec Diff、语义编译、执行验证、消费者验证。
- 保存审计线索：字段确认、Proposal 生成、发布预演、最终发布。

它不负责：

- 不成为语义中心里的正式语义对象。
- 不改变 `Cube / Ontology / Binding` 的定义。
- 不进入 `Data Agent / BI / 数据分析` 的消费契约。
- 不作为查询、编译或权限判断的语义条件。

### 4.2 Asset Package

`Asset Package` 是项目里的待建设资产单元，可以来自批量扫描、AI 推荐、人工选择或 Release 修订回流。

常见类型：

- 事实主题候选
- 维度主题候选
- 指标组候选
- 业务对象绑定组

每个资产包都必须包含两层语义和绑定关系：

```text
Asset Package
  -> Cube Layer
  -> Ontology Layer
  -> Binding
```

### 4.3 Field Candidate

`Field Candidate` 是数仓建模工程师冷启动时的核心审阅对象。字段候选行至少表达：

- 物理字段与来源表
- 建议语义名
- 字段角色：维度、度量、时间、属性、主键、外键
- Cube 映射：dimension、measure、time、join key
- 本体锚定：object、property、metric、relation、action
- 证据、置信度、风险
- 操作：采纳、改写、忽略、合并、退回重生成

### 4.4 Proposal

`Proposal` 是准备发布到语义中心的变更草案，不是正式语义资产。

它包含：

- 字段采纳结果
- Cube 变更
- 本体变更
- Binding 变更
- Spec Diff
- 影响范围
- 发布前风险

### 4.5 Release

`Release` 是语义中心的正式版本结果。发布后：

- Data Agent 可消费
- BI 可消费
- 数据分析可消费
- 报表和业务应用可消费

下游消费者只认语义中心 Release，不依赖工作台、Build Project 或 Asset Package。

## 5. 两层语义设计

### 5.1 Cube Layer：分析执行真相源

Cube 层回答：

- 数据从哪张物理表来？
- 粒度是什么？
- 哪些字段是维度、度量、时间？
- 默认聚合、过滤条件、Join 怎么定义？
- 是否能被语义中心编译成可执行 SQL？

Cube 层偏向数仓建模工程师熟悉的分析模型、事实表/维表、指标计算和查询执行。

### 5.2 Ontology Layer：业务语义真相源

本体层回答：

- 这个资产对应什么业务对象？
- 字段在业务上叫什么？
- 指标口径如何被业务理解？
- 对象之间有什么业务关系？
- 消费者应该怎么理解这些语义？

本体层偏向业务对象、属性、指标、关系、动作、术语和治理口径。

### 5.3 Binding：两层连接

Binding 是两层语义之间的结构化连接：

- `BusinessObject -> Cube`
- `BusinessProperty -> Dimension`
- `BusinessMetric -> Measure`
- `BusinessRelation -> Join / Semantic Relation`
- `BusinessAction -> Event Cube / Fact`

冷启动阶段只做轻本体锚定，不做完整本体治理。复杂术语生命周期、跨域关系图谱和审批流应留给后续治理工作台。

## 6. 首屏产品形态

首屏不是项目管理列表，也不是纯聊天页，而是：

> 推荐建设范围 + 用户确认 + 候选队列预览/生成。

### 6.1 默认结构

左侧：建设范围确认。

- 业务域
- 源表范围
- 建设策略
- 是否对齐已有语义资产
- AI 推荐补全范围

右侧：候选资产队列。

- 候选事实主题
- 候选维度主题
- 候选指标组
- 候选对象绑定
- 风险、置信度、状态和主动作

### 6.2 Product Review 修正

首屏不能做成从零填写的重表单。更好的体验是：

```text
系统识别/推荐一个建设范围
  -> 用户确认或微调
  -> 预览候选队列
  -> 生成 Build Project
```

AI 在这里的价值是补全范围、解释候选、提示缺口，而不是把用户带入开放聊天。

### 6.3 项目列表的位置

已有项目可以作为辅助入口：

- 有未完成项目时，在首屏顶部提示“继续上次建设”。
- 项目列表作为侧栏或次入口存在。
- 不让项目管理状态看板抢占首屏主任务。

## 7. 单资产画布

进入某个 `Asset Package` 后，主画布应以字段候选表为主，而不是聊天流、Cube 编辑器或本体编辑器。

推荐结构：

```text
左侧：资产上下文
中间：字段候选主画布
右侧：AI 证据与发布检查
```

### 7.1 左侧：资产上下文

只展示上下文，不承担复杂操作：

- 所属 Build Project
- 资产标题
- 来源表
- 粒度
- 风险
- 当前状态
- 已有语义覆盖情况

### 7.2 中间：字段候选主画布

默认阶段是字段审阅。Cube、本体和 Binding 通过字段行内映射呈现。

字段行包括：

- 物理字段
- 建议语义名
- 字段角色
- Cube 映射
- 本体锚定
- 证据与风险
- 操作

### 7.3 阶段化展示

避免一开始并列展示太多 tab。单资产画布按阶段推进：

1. `字段审阅阶段`
   - 主画布是字段候选表。
   - Cube、本体、Binding 是字段行上的映射列。

2. `草案确认阶段`
   - 用户采纳足够字段后，系统生成 Proposal。
   - 展示 Cube 摘要、本体摘要、Binding 摘要。

3. `发布预演阶段`
   - 展示 Spec Diff、语义中心编译、影响分析、执行验证、消费者验证。

### 7.4 右侧：AI 证据与发布检查

右侧不是聊天主舞台，而是上下文助手：

- 当前字段证据
- 冲突说明
- 建议修复 patch
- 发布检查状态
- 编译失败、执行验证失败和消费者验证失败解释

可以提供输入框，但不以聊天流作为主工作面。

## 8. AI 定位

AI 的定位是：

> 强生成，弱主语；强建议，弱决策；强加速，弱真值。

### 8.1 项目级 AI

在 `Build Project` 创建和范围确认阶段：

- 根据业务域推荐源表范围。
- 根据源表、已有语义资产、查询日志生成候选 Asset Package。
- 识别覆盖缺口、高风险候选、重复候选。
- 解释为什么推荐这些候选。

输出是候选队列，不是正式语义。

### 8.2 资产级 AI

在单个 `Asset Package` 审阅阶段：

- 推荐字段语义名、字段角色、默认聚合。
- 推荐 Cube 映射和轻本体锚定。
- 解释字段画像、血缘、已有语义冲突。
- 生成可审阅 patch。

输出是建议和 patch，不是直接发布。

### 8.3 发布级 AI

在 Proposal / Release 阶段：

- 解释 Spec Diff。
- 解释语义中心编译失败。
- 解释执行验证失败。
- 解释消费者验证失败。
- 给出修复建议。

输出是诊断和修复建议，不是绕过门禁。

### 8.4 AI 体验护栏

所有 AI 建议必须：

- 有证据来源。
- 有置信度或风险等级。
- 可批量采纳，但高风险不可批量采纳。
- 可撤销。
- 形成 diff，而不是静默改写。
- 不绕过人工确认。

产品文案中不使用 `Copilot` 作为主导航名或页面名。可以使用“AI 建议”“证据助手”“智能补全”等局部能力名。

## 9. 发布与生命周期

发布链路固定为：

```text
Asset Package 审阅完成
  -> Proposal
  -> 语义中心编译预演
  -> Gateway 物理 SQL dry-run
  -> Release
  -> 消费者可用
```

### 9.1 语义中心职责

语义中心负责：

- 持久化正式语义资产。
- 封装语义到 SQL / Query Plan 的编译能力。
- 生成 release diff 和影响分析。
- 判断语义资产是否具备发布条件。

产品文案中主界面表达为“语义编译”或“可发布检查”，详情中说明来自语义中心。

### 9.2 Gateway 职责

Gateway 只做执行面验证。

它只接收：

- 物理 SQL
- 执行上下文
- 权限和 guardrail 所需信息

它不接收：

- Semantic Spec
- Build Project
- Asset Package
- 业务口径
- 本体定义

产品上表达为“执行验证”或“物理 SQL dry-run”，不要表达为“语义验证”。

### 9.3 消费者职责

Data Agent、BI、数据分析、报表和业务应用都只消费语义中心 Release。

它们不直接依赖：

- 建模工作台
- Build Project
- Asset Package
- AI 建议

### 9.4 UI 表达

主界面统一叫“可发布检查”，下面分四类状态：

1. `语义草案完整性`
   - 是否已生成 Cube / 本体 / Binding，必填口径是否齐全。

2. `语义编译`
   - 是否能被语义中心编译成可查询表达。

3. `执行验证`
   - 物理 SQL dry-run 是否通过，详情中说明由 Gateway 执行。

4. `消费者可用性`
   - Data Agent、BI、数据分析的样例问题或消费契约是否通过。

每个检查项必须回答：

- 当前是否可发布？
- 为什么不可发布？
- 下一步怎么修？

## 10. 生命周期

`Build Project` 长期保留，但它保存的是建设过程、审阅证据和发布历史，不作为消费者可见语义。

推荐状态：

- `Draft`：选择业务域、源表范围和建设策略。
- `Scanned`：生成候选 Asset Package 队列。
- `Reviewing`：逐项审阅字段证据、Cube 和轻本体锚定。
- `Validated`：完成 Spec Diff、编译预演和执行面验证。
- `Released`：发布到语义中心，形成正式 Release。
- `Maintaining`：从项目历史发起修订、补齐和治理回流。

长期保留的价值：

- 回看当时为什么这么建。
- 回看字段证据和 AI 建议。
- 追踪谁确认了哪些字段。
- 从 Release 反向发起修订。
- 支持后续补齐、修复和治理回流。

## 11. 成功指标

### 11.1 冷启动效率

- 从选择范围到生成候选队列的时间。
- 从候选资产进入画布到生成 Proposal 的时间。
- 字段候选默认生成覆盖率。

### 11.2 建模质量

- 字段采纳率。
- 高风险字段人工处理率。
- 语义编译通过率。
- 发布前被发现的冲突数量。

### 11.3 用户心智

- 用户能否清楚说出当前是在 Build Project、Asset Package、Proposal 还是 Release 阶段。
- 用户是否理解 AI 是建议来源，而不是语义真值。
- 用户是否理解发布目标是语义中心，而不是 Data Agent 或 Gateway。

### 11.4 长期维护

- Release 后从历史项目发起修订的次数。
- 未完成项目继续推进率。
- 已发布资产的治理回流闭环率。

## 12. 非目标

本设计不包含：

- 完整本体图谱编辑器。
- 复杂审批流。
- 自动批量发布。
- 后台无人值守 Agent 建模。
- Gateway 接收语义 Spec 或 Query Plan 的新能力。
- Data Agent 直接依赖建模工作台。

## 13. 风险与缓解

### 风险 1：Build Project 变成沉重项目管理页

缓解：

- 首屏以推荐范围和候选队列为主，不以状态看板为主。
- 项目列表降为次入口。
- 默认引导用户完成下一步行动。

### 风险 2：AI 被压得太弱

缓解：

- AI 默认主动生成范围建议、候选队列和字段候选。
- 用户审阅和采纳，而不是从零填写。
- AI 输出必须结构化落到候选、字段建议、patch 和诊断中。

### 风险 3：两层语义增加心智负担

缓解：

- 字段候选表作为主画布。
- Cube、本体、Binding 行内呈现。
- 详细 Cube / 本体摘要只在草案确认阶段展开。

### 风险 4：发布边界再次漂移

缓解：

- 主界面只说“可发布检查”。
- 详情中明确语义中心编译、Gateway 执行验证和消费者验证。
- Data Agent / BI 只作为消费者验证对象，不作为发布目标。

## 14. 结论

最终产品形态应固定为：

> 语义建设工作台是长期 Build Project 容器。它通过 AI 加速范围确认、候选生成和字段审阅，通过字段候选主画布统一建设 Cube 层、本体层和 Binding，通过 Proposal 与可发布检查发布到语义中心。Data Agent、BI、数据分析等下游只消费语义中心 Release。

这套形态既保留 A 的规范性，也吸收 C 的批量能力和 B 的 AI 冷启动价值，同时避免三者作为平级产品入口继续摇摆。
