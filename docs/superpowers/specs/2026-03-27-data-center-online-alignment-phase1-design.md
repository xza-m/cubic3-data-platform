# 数据中心线上联调第一阶段设计

> 日期：2026-03-27
> 主题：按线上环境和后端真实能力，将 Pencil 新界面落成数据中心第一阶段可交付前端

## 1. 背景

当前仓库在前端层同时存在三套“真相”：

1. 旧前端页面与旧交互
2. 基于 Pencil 设计稿实现出的新界面
3. 后端当前真实可用的接口与运行链路

这三者没有统一收口，导致页面表现与真实能力脱节，具体包括：

- 新界面没有直接建立在旧页面已验证的真实逻辑上
- Pencil 页面中残留 mock 数据、假交互或未联调的模块
- 路由入口与真实可用页面不一致，用户难以判断哪个页面才是正式入口

本设计的目标不是“再做一套新前端”，而是把 Pencil 新界面真正收口到后端真实能力上，并形成可分阶段替换整个平台的方法。

## 2. 设计前提

本次设计以用户确认的约束为准：

- **唯一真相**：以后端现有接口能力为准
- **总体策略**：整包替换，但必须分阶段推进
- **第一阶段目标**：完整打通数据中心闭环
- **对后端暂未支持的 Pencil 能力**：保留布局，但必须禁用并说明原因

据此，任何 Pencil 中存在但后端没有稳定契约的能力，都不能通过 mock 数据、本地假状态或“看起来能点”的方式上线。

## 3. 目标与非目标

### 3.1 目标

第一阶段要交付一套数据中心新界面，满足以下要求：

- Pencil 成为数据中心默认视觉基线
- 数据中心所有核心动作都绑定真实后端接口
- 所有不可用能力都进入明确禁用态
- 旧页面中已稳定的能力不因替换而丢失
- 有配套测试、联调路径和文档更新

### 3.2 非目标

第一阶段不处理以下内容：

- 查询中心、语义中心、平台壳的大规模功能替换
- 为了适配设计稿而新增后端接口
- 用本地状态或 mock 数据补齐后端未实现能力
- 无关页面的样式统一与大规模组件重构

## 4. 方案比较

### 方案 A：后端真相驱动的分阶段联调

做法：

- 先盘点后端能力、旧页面能力和 Pencil 模块
- 用“真实能力矩阵”驱动 Pencil 页面落地
- 原地替换旧路由页面，不保留双轨前端
- 未支持能力统一禁用

优点：

- 风险最低，符合 KISS
- 不制造第二套假前端
- 每一阶段都可以形成真实可交付闭环

缺点：

- Pencil 稿需要收缩，第一版不会 100% 还原理想交互

### 方案 B：设计稿优先的大面积替换

做法：

- 优先还原 Pencil 页面
- 后端联调和能力补齐后置

优点：

- 视觉统一快

缺点：

- 极易再次制造“页面像上线但不可用”的状态
- 明显违背 KISS 与 YAGNI

### 方案 C：增加前端适配层兼容新旧页面

做法：

- 在现有页面与接口之间增加更厚的前端适配层
- 让 Pencil 页面和旧页面都走适配层

优点：

- 短期兼容性较好

缺点：

- 引入额外复杂度
- 容易演变成长期技术债

### 推荐结论

采用 **方案 A：后端真相驱动的分阶段联调**。

## 5. 第一阶段范围

第一阶段固定为**完整数据中心闭环**，页面与动作范围如下：

### 5.1 数据源

- 数据源列表
- 数据源创建
- 数据源编辑
- 测试连接
- 同步目录
- 列表筛选与状态展示

### 5.2 数据集

- 数据集列表
- 数据集详情
- 数据集预览
- 数据集注册
  - `physical`
  - `virtual`
  - `file`

### 5.3 阶段完成标准

第一阶段完成时必须同时满足：

1. Pencil 新界面成为数据中心默认入口和默认视觉基线
2. 所有可点击动作都绑定真实后端请求或明确禁用态
3. 旧页面里仍被真实使用的能力没有丢失
4. 数据中心的测试、文档和联调路径可以证明这是正式实现，而不是设计稿壳子

## 6. 页面替换策略

### 6.1 总体替换原则

- 路由不新开双轨，优先在现有数据中心路由上原地替换
- 旧页面真实逻辑优先复用
- Pencil 负责视觉、信息架构和布局
- 后端不支持的能力统一进入禁用态

### 6.2 页面级策略

#### 数据源列表页

- 直接替换为 Pencil 风格布局
- 数据继续来自现有 `/api/v1/data-center/datasources`
- 保留真实动作：筛选、创建、编辑、删除、测试连接、同步目录、统计摘要
- Pencil 中未支持的高级治理模块展示为禁用态

#### 数据源创建/编辑

- 视觉切换到新界面
- 继续复用现有字段模型、校验与提交契约
- 不新增后端没有的配置字段语义

#### 数据集列表页

- 替换为新界面
- 绑定现有列表、筛选、跳详情、跳注册链路
- 若 Pencil 有卡片化/多视图切换，可保留低风险部分

#### 数据集详情页

- 作为第一阶段重点页
- 必须展示真实元数据、来源信息、字段摘要、更新时间、预览入口
- 血缘、质量、影响分析等能力如后端未支持，则展示禁用模块

#### 数据集注册页

- `physical / virtual / file` 三类流程都保留现有提交语义
- 新界面可重排步骤，不改变接口契约
- `virtual` 必须继续基于真实 SQL 与真实 source 语义

## 7. 能力矩阵

### 7.1 必须真实可用

- 数据源列表加载、分页、基础筛选、状态展示
- 数据源创建/编辑
- 测试连接
- 同步目录
- 数据集列表
- 数据集详情
- 数据集预览
- 三类数据集注册：`physical / virtual / file`

### 7.2 允许保留但必须禁用

- 高级治理看板
- 血缘关系图
- 影响分析
- 质量评分/规则中心
- 批量运维动作
- 高级目录编排
- 自动推荐/智能修复能力

### 7.3 严禁出现

- mock 数据冒充真实数据
- 按钮可点但没有请求
- 请求失败却展示成功态
- 新页面丢失旧页面真实能力
- 同一动作在不同页面走不同接口语义

## 8. 状态与错误处理

### 8.1 页面级状态

每个页面只允许进入以下明确状态：

- `loading`
- `ready`
- `empty`
- `error`
- `disabled-section`

### 8.2 行为约束

- 首屏加载只能展示 skeleton/占位，不展示假数据
- 空态必须区分“无数据”和“筛选后无结果”
- 局部失败只影响局部模块，不拖垮整页
- 所有动作型操作都要有 `pending / success / error` 三态
- 提交失败后保留表单输入，不清空用户上下文

### 8.3 长耗时任务

目录同步类异步动作按真实后端能力处理：

- 若只有任务提交结果，则显示“已进入队列/已提交”
- 若没有真实进度接口，不伪造进度条
- Pencil 中理想化实时进度能力在第一阶段不补做

### 8.4 禁用态规范

对后端暂不支持的模块统一处理：

- 布局保留
- 控件不可点击
- 明示文案：`当前阶段未接入后端能力`
- tooltip 或说明文案解释原因
- 不显示假数字、假图表、假列表

## 9. 页面与组件拆分

### 9.1 页面层职责

第一阶段维持现有数据中心主路由，仅原地替换：

- 数据源列表页
- 数据集列表页
- 数据集详情页
- 数据集注册页
- 文件型数据集注册页（若现有流程独立）

页面层只负责：

- 路由参数
- 页面级状态切换
- 组合子组件
- 调用页面级 hook

### 9.2 推荐抽取的公共组件

- `DataCenterPageShell`
- `CapabilityGateCard`
- `ResourceSummaryPanel`
- `ConnectionStatusBlock`
- `PreviewPanel`
- `RegisterFlowShell`

### 9.3 边界约束

- 页面负责组织，不直接拼装接口细节
- hook 负责列表、详情、预览、提交和异步动作
- 展示组件只吃 props，不直接请求接口
- 禁用模块不能偷偷带本地 mock 状态

## 10. 联调推进顺序

### Wave 0：契约盘点与替换清单

输出一张真实能力映射表：

- 后端接口清单
- 旧页面真实能力清单
- Pencil 页面模块清单
- 映射关系：`已支持 / 需复用旧逻辑 / 必须禁用`

### Wave 1：数据源页先收口

落地：

- 数据源列表
- 创建/编辑
- 测试连接
- 同步目录
- 筛选与状态展示

### Wave 2：数据集列表与详情

落地：

- 数据集列表
- 数据集详情
- 字段摘要
- 预览
- 来源信息

### Wave 3：三类数据集注册

落地：

- `physical`
- `virtual`
- `file`

### Wave 4：旧入口清理与统一禁用态

收尾：

- 清理残余旧入口
- 统一禁用态组件与文案
- 检查 mock 数据残留
- 检查新旧页面跳转断裂

## 11. 验证策略

### 11.1 页面级单测

重点验证：

- 请求参数正确
- 加载/空态/失败态/禁用态正确
- 表单提交语义正确
- 旧链路入口仍然通畅

### 11.2 联调级页面回归

至少覆盖：

- 数据源页打开
- 创建/编辑
- 测试连接
- 同步目录
- 数据集列表与详情
- 预览
- 三类注册流程

### 11.3 路由与入口回归

确保平台壳中的数据中心入口全部落到新实现，不跳回旧壳子或孤立设计稿页面。

### 11.4 人工联调验收

必须人工确认：

- 没有 mock 数据残留
- 没有“可点击但无请求”的假交互
- 禁用模块说明清晰
- Pencil 已经成为默认视觉基线，而不是另一路未接线页面

## 12. 文档更新要求

第一阶段完成后，至少检查以下基线文档是否需要同步：

- `README.md`
- `docs/TECH_STACK_AND_ARCHITECTURE.md`
- `docs/STARTUP_GUIDE.md`
- `docs/DOC_ALIGNMENT_REPORT.md`
- `frontend/README.md`

若默认入口、页面结构、验证路径发生变化，必须更新这些文档。

## 13. 风险与缓解

### 风险 1：Pencil 稿理想能力过多，联调过程中继续扩 scope

缓解：

- 用真实能力矩阵做唯一判断依据
- 所有未支持能力只能禁用，不允许临时 mock

### 风险 2：旧页面逻辑散落，原地替换时容易误删真实能力

缓解：

- Wave 0 先列完整动作清单
- 以页面级回归和人工联调兜底

### 风险 3：为了视觉统一而重写过多页面逻辑

缓解：

- 先替换壳层与布局，再复用旧逻辑
- 限制组件抽取范围，不做无关重构

## 14. 结论

第一阶段应交付的是：

> 一套以后端现有能力为真相、以 Pencil 为默认视觉基线、具备真实联调与回归验证能力的数据中心新前端实现。

这意味着第一阶段不是把所有理想功能做完，而是确保所有已经展示给用户的功能都真实可信、可验证、可继续演进。
