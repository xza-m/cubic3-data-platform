# 本体工作台对象聚合根重构设计（v2）

> 日期：2026-04-14
> 主题：将 `/semantic/ontology` 从“资产并列管理台”重构为“对象聚合根主工作台 + 专项索引辅助治理 + Cube 分层承载技术实现”

## 1. 背景

当前本体工作台把 `对象 / 属性 / 业务指标 / 关系 / 动作 / 术语 / 权限` 作为并列顶级资产入口。这个结构虽然便于快速拼出 CRUD，但它存在两个根本问题：

1. 语义建模上的一等公民，被机械映射成了产品交互上的一级入口。
2. 本体层的业务语义聚合根被拆碎，用户看到的是“资产类型大全”，而不是“业务对象世界”。

这会直接带来几类后果：

- 用户进入页面后，需要先理解“我要看对象、关系还是动作”，而不是先定位“我在建模哪个业务对象”。
- 属性、关系、动作、规则等原本附着于对象的能力，被抬成了并列顶级导航，认知负担高。
- 业务指标同时横跨业务语义与分析实现，但当前模型没有把这种双重属性表达清楚。
- 页面会自然演变成“大一统工作台”，把定义、验证、治理都塞在一处，职责边界模糊。

因此，本次改版不再把“是否显式绑定 Cube”作为唯一中心问题，而是先回到更本质的一层：

`本体工作台的产品 IA，必须先匹配语义建模的聚合根。`

## 2. 设计结论

本体工作台正式采用以下整体方向：

`对象聚合根主工作台 + 专项索引辅助治理 + Cube 分层承载技术实现`

这意味着：

- 本体层以 `业务对象 / 稳定语义主题` 作为主聚合根。
- Cube 层以 `分析主题 Cube` 作为主聚合根。
- 属性、关系、动作、状态、规则不是一级入口，而是对象下的语义能力。
- 业务指标不再作为本体层独立顶级资产，而采用“语义声明在对象、执行实现在 Cube”的混合模式。

## 3. 目标与非目标

### 3.1 目标

- 让用户进入本体后，先看到对象，再看到对象的语义能力。
- 把本体层与 Cube 层的职责边界明确下来，避免职责重叠与双写。
- 降低主导航复杂度，同时保留跨对象治理的全局视角。
- 为后续验证、治理、发布提供更稳定的对象上下文。

### 3.2 非目标

- 当前阶段不把 `对象 + 语义主题` 双聚合根同时做成前台主入口。
- 当前阶段不把辅助视图扩展成第二套完整编辑工作台。
- 当前阶段不让本体层承载 SQL、聚合公式、性能策略等分析实现细节。

## 4. 设计原则

### 4.1 KISS

主入口只保留“对象列表”，不再暴露七类资产并列导航。复杂能力通过对象详情内的分层结构和少量辅助视图承载。

### 4.2 YAGNI

第一阶段只支持“对象聚合根”为主入口，不提前做“对象 + 主题”双轴导航，也不提前做全局复杂编排。

### 4.3 SOLID

- 对象详情页负责表达对象语义全貌。
- 辅助视图负责跨对象检索、筛选、风险发现。
- Cube 工作台继续负责分析实现。

不同职责落在不同页面上下文中，避免继续堆成一个超级页面。

### 4.4 DRY

- 本体层不重复承载 Cube 的公式和执行逻辑。
- 辅助视图不复制对象详情的完整编辑能力。
- 业务指标的业务语义与技术实现分层承载，避免双份技术定义。

## 5. 新的信息架构

### 5.1 顶层结构

```text
Ontology
├─ 对象列表                     ← 主入口 / 主建模入口
│  ├─ Student
│  ├─ KnowledgePoint
│  ├─ CourseUnit
│  └─ ...
└─ 辅助视图                     ← 治理与检索入口，不是主建模入口
   ├─ 关系索引
   ├─ 规则索引
   └─ 业务指标索引
```

### 5.2 废弃的顶级入口

以下资产类型不再作为主导航：

- 属性
- 业务指标
- 关系
- 动作
- 术语
- 权限

它们将分别回归：

- 属性、关系、动作、状态、规则：对象详情页
- 术语/别名：对象定义或业务指标口径
- 权限：规则与治理区

## 6. 对象详情页设计

对象详情页不应再是“对象下面再堆八个平级 tab”，而应表达对象语义全貌。

```text
对象详情
├─ 顶部上下文
├─ 对象定义
├─ 对象能力
├─ 对象关联
└─ 治理与生命周期
```

### 6.1 顶部上下文

作用：回答“我现在在看哪个对象，以及它当前处于什么状态”。

包含：

- 对象名
- 状态
- 简介
- 保存 / 发布
- 最近修改时间
- 关系数、规则数、关联 Cube 数、指标数

### 6.2 对象定义

作用：定义对象是什么，而不是怎么执行。

包含：

- 名称、标题、描述、别名
- 对象分类
- 语义边界说明
- 术语信息

### 6.3 对象能力

作用：表达对象本身具备哪些静态与动态能力。

结构：

- 静态能力：属性、状态
- 动态能力：动作、规则

### 6.4 对象关联

作用：表达对象与外部世界的连接关系。

包含：

- 关系
- 业务指标
- 关联 Cube

### 6.5 治理与生命周期

作用：表达对象是否可发布、是否存在风险、历史如何演进。

包含：

- 校验结果
- 发布检查
- 历史记录
- 审计记录
- 绑定风险

## 7. 对象详情页的分层与折叠策略

这是 `Claude` 评审指出的关键风险点：如果把上述五个区块全部平铺，页面会迅速演变成“上帝页面”。

因此，本方案明确加入以下约束：

### 7.1 默认展开

- 顶部上下文
- 对象定义摘要
- 当前风险摘要

### 7.2 默认折叠或摘要化

- 属性列表：默认展示前 `5` 项 + 总数
- 关系列表：默认展示前 `5` 项 + “查看全部”
- 动作列表：默认展示前 `5` 项
- 规则列表：默认展示前 `5` 项
- 历史记录 / 审计记录：默认收起

### 7.3 进入二级子页或抽屉的内容

- 大量属性
- 大量关系
- 全量规则
- 全量业务指标
- 历史与审计明细

### 7.4 页面组织原则

对象详情页展示“全貌与上下文”，而不是承载所有明细的完整展开。

也就是说：

- 对象详情页是对象语义主页
- 明细列表页/抽屉页是对象的二级延展视图

## 8. 业务指标的正式定义

这是 `Gemini` 评审指出的核心概念张力：业务指标既被称为对象能力，又被称为桥梁层，如果不下正式定义，后续实现会摇摆。

本方案将业务指标定义为：

`对象下的语义能力声明，不是对象内部的完整技术实现。`

这一定义意味着：

- 在对象语义层，业务指标描述“该对象有哪些关键业务度量，以及这些度量在业务上是什么意思”。
- 在 Cube 分析层，业务指标的计算方式、聚合方式、Join 依赖、执行性能策略才被真正落实。

### 8.1 对象内保留的内容

- 指标名称
- 业务口径
- 解释文案
- 适用范围
  - 这里特指业务适用范围，例如“课堂学习场景”“章节完成结果”“阶段性诊断分析”
  - 不表示技术执行范围，不表示“可在哪些 Cube 上运行”
- 绑定到哪些 Cube / Measure
- 当前绑定状态

### 8.2 Cube 内承载的内容

- 计算公式
- 聚合方式
- Join 依赖
- 性能策略
- 执行实现

### 8.3 设计含义

业务指标不是本体层的独立顶级资产，也不是对象内部的完整技术块，而是：

`对象语义声明 + Cube 技术实现` 的桥梁。

## 9. 业务指标一致性保障机制

这是两边评审都指出的风险：如果对象中的业务口径与 Cube 中的技术实现发生漂移，整个语义层会失去可信度。

因此，本方案要求增加一套最小一致性保障机制：

### 9.1 绑定状态

所有业务指标绑定都必须有明确状态：

- `fresh`：绑定正常
- `warning`：绑定可用但需要人工确认
- `stale`：绑定失效

### 9.2 触发检查的时机

- Cube Measure 改名
- Cube Measure 删除
- Join 路径变化
- 对象与 Cube 解绑
- 业务口径修改

### 9.3 必需能力

- 发布前校验
- 影响分析
- 变更提示
- 历史版本比对

### 9.4 绑定异常处理路径

当业务指标绑定状态变为 `warning` 或 `stale` 时，系统必须给出明确、可闭环的处理路径，而不是只在后台记录状态。

#### 提示出现位置

- 对象详情页顶部上下文：显示风险 badge，例如“2 个指标绑定异常”
- 对象详情页“治理与生命周期”区：显示绑定风险摘要
- 业务指标列表摘要区：对具体指标显示 `warning / stale` 状态标记

#### 用户进入修复的路径

- 点击顶部风险 badge
- 点击治理区的绑定风险卡片
- 点击业务指标列表中的异常状态标记

三种入口都进入同一个“绑定修复抽屉”或对象详情页内联展开区，而不是跳到辅助视图中完成修复。

#### 在修复视图中允许的操作

- 重新绑定到新的 `Cube / Measure`
- 查看影响分析
- 查看历史绑定版本
- 跳转到关联 Cube 查看技术实现

当前阶段不提供：

- 自动修复
- 在辅助视图中直接深度编辑
- 跨对象批量改写口径

#### 修复完成后的状态回写

- 重新绑定成功且校验通过：状态回到 `fresh`
- 重新绑定成功但仍有人工确认项：状态保持 `warning`
- 未完成修复或校验失败：状态保持 `stale`

状态变化必须同时回写到：

- 对象详情页顶部上下文
- 对象内业务指标摘要区
- 治理与生命周期区

### 9.5 不做的事情

当前阶段不要求自动修复绑定，只要求做到：

- 能发现
- 能提示
- 能阻断错误发布

这符合 `YAGNI`，也足够支撑第一阶段的可靠性。

## 10. 辅助视图设计

辅助视图保留，但它们的定位必须非常克制。

## 10.1 关系索引

作用：

- 查看跨对象关系网络
- 筛选失效关系
- 按源对象/目标对象检索
- 跳转回对象详情

## 10.2 规则索引

作用：

- 查看对象规则分布
- 筛选阻断发布或冲突规则
- 查看风险摘要
- 跳转回对象详情

## 10.3 业务指标索引

作用：

- 查看对象下的业务指标
- 筛选未绑定 / 失效 / 跨 Cube 指标
- 查看口径摘要
- 跳转回对象详情中的指标区

## 10.4 辅助视图边界铁律

这是本方案必须守住的产品约束：

1. 只做检索、筛选、风险发现、跳转。
2. 不承载完整编辑流。
3. 不与对象列表并列为一级主导航。
4. 深度编辑统一回对象详情页。

### 10.5 最容易失控的点

`业务指标索引`

原因：

- 指标天然是高频查看与修改对象
- 一旦在索引页里放开完整编辑，平台会迅速长出第二套指标编辑入口

所以这一页必须重点守住：

- 可看摘要
- 可看风险
- 可跳转
- 但不直接承担完整维护

## 11. Ontology 与 Cube 的职责边界

### 11.1 Ontology 负责

- 业务对象语义
- 属性、关系、动作、状态、规则
- 业务指标口径
- 对象与 Cube 的显式绑定关系
- 治理、审计、发布语义

### 11.2 Cube 负责

- Dimension
- Measure
- Join
- Segment
- 聚合逻辑
- 执行与性能实现

### 11.3 一句话边界

`Ontology 解释业务世界，Cube 落实分析世界。`

## 12. 用户操作主流程

### 12.1 主流程

```text
进入本体
→ 查看对象列表
→ 进入某个对象
→ 完成对象定义
→ 补充属性 / 状态 / 动作 / 规则
→ 配置关系与业务指标
→ 绑定关联 Cube
→ 运行校验
→ 发布
```

### 12.2 辅助治理流程

```text
进入关系/规则/业务指标索引
→ 搜索或筛选问题项
→ 查看摘要
→ 跳回对象详情处理
```

## 13. 风险与缓解

### 风险 1：对象详情页仍然过重

缓解：

- 强制使用摘要 + 折叠 + 子页/抽屉策略
- 不把明细列表全部平铺在首页

### 风险 2：业务指标语义与实现漂移

缓解：

- 引入绑定状态
- 在发布前做一致性校验
- 对 Cube 变更做影响提示

### 风险 3：辅助视图长成第二套工作台

缓解：

- 明确“只检索不深编”的边界
- 所有深度编辑回对象详情

## 14. 推荐实施顺序

1. 先重写本体工作台 IA，切换到对象列表主入口
2. 重做对象详情页结构
3. 加入对象详情页的折叠/摘要策略
4. 重构业务指标展示为“语义声明 + 技术绑定摘要”
5. 补关系/规则/业务指标索引页
6. 最后重接校验、审计、发布链路

## 15. 结论

本体工作台不应再继续沿用“资产并列管理台”的模式。

正式方向应为：

`以对象为中心的语义建模工作台`

其中：

- 对象是主聚合根
- 辅助视图是治理工具，不是第二主入口
- 业务指标是对象下的语义声明，并通过 Cube 实现落地
- Ontology 与 Cube 分层清晰协作，而不是互相吞并

这版方案吸收了本轮 `Claude` 与 `Gemini` 评审意见，较前一版更适合作为后续实现与视觉设计的基线。
