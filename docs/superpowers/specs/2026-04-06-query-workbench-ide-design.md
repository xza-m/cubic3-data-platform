# 查询工作台 IDE 化与查询发布任务设计

> 日期：2026-04-06
> 主题：将查询中心重构为 IDE 式多 Tab SQL 工作台，并建立“先保存查询，再发布为任务”的事件流

## 1. 背景

当前查询中心已经完成了第一轮平台化收敛，但仍存在三个关键体验问题：

1. 查询页仍以单编辑器为主，无法像 SQLLab 一样同时处理多个 SQL 工作上下文
2. 查询资产与查询任务的关系不清晰，用户缺少“先保存查询、再发布为任务”的自然路径
3. `我的查询 / 历史 / 模板 / 定时查询` 仍保留较强的平行页面心智，削弱了 `/queries` 作为查询主工作台的定位

用户已经明确提出两点新增目标：

- 查询去掉无效提示条，收敛为更直接的工作台
- 查询支持多 SQLLab Tab，并支持将查询先保存为资产，再发布为任务

同时，用户明确确认了以下关键约束：

- 查询工作台采用 **IDE 式设计**
- 多 Tab 需要 **持久化到后端，跨刷新、跨设备恢复**
- 每个 Tab **独立绑定数据源、SQL、结果和保存状态**
- 事件流应为：`先保存为查询，再发布为任务`

## 2. 设计前提

本设计遵循当前平台边界：

- `/queries` 是查询开发工作台
- `/queries/scheduled` 是查询任务管理页，不承担 SQL 编辑职责
- 查询资产与任务实体不是同一个概念
- 本轮不重做后端任务系统，只复用现有查询、执行、任务相关能力

工程原则约束如下：

- `KISS`：不在查询页内再引入复杂调度编排器
- `YAGNI`：本轮先解决多 Tab、保存查询、发布任务，不扩到 notebook 或多语言工作流
- `SOLID`：Tab 是编辑上下文，查询是资产，任务是运行实体，三者职责分层
- `DRY`：`我的查询 / 历史 / 模板` 不再各自重复承担编辑能力，而是回流到统一工作台

## 3. 目标与非目标

### 3.1 目标

- 将 `/queries` 重构为 IDE 式多 Tab SQL 工作台
- 支持多 Tab 持久化恢复，跨刷新、跨设备保留上下文
- 支持每个 Tab 独立绑定数据源、SQL、执行结果、保存状态和查询 ID
- 明确建立 `保存查询 -> 发布为任务` 的事件流
- 将 `我的查询 / 历史 / 模板` 收口为工作台内资产面板
- 保留 `/queries/scheduled` 作为任务管理页

### 3.2 非目标

- 当前阶段不把查询工作台扩展为 notebook、脚本、SQL 混合多语言平台
- 当前阶段不在查询页内直接做复杂 cron 编排、审批流或项目目录发布
- 当前阶段不废弃 `/queries/scheduled` 的独立任务管理职责
- 当前阶段不改变现有后端查询执行接口的基本语义

## 4. 方案比较

### 方案 A：轻量多 Tab 工作台

做法：

- 在现有查询页上直接增加多个本地 Tab
- 每个 Tab 支持独立 SQL 和结果
- 保存查询与发布任务仍主要依赖弹窗

优点：

- 实现成本低
- 对现有代码侵入最小

缺点：

- 若仅做前端本地状态，无法满足跨刷新、跨设备恢复
- 查询资产、任务实体、工作上下文的边界仍然较弱

### 方案 B：IDE 式查询工作台

做法：

- `/queries` 成为统一 SQL 工作台
- 顶部多 Tab，每个 Tab 独立持久化
- 左侧收口资源、我的查询、模板、历史
- 工具栏仅保留核心动作：运行、停止、格式化、保存查询、发布为任务
- 任务运行管理继续放在 `/queries/scheduled`

优点：

- 最符合用户对 SQLLab 的预期
- 符合 `SOLID`，编辑上下文、查询资产、任务运行实体边界清晰
- 符合 `DRY`，避免多个查询子页重复承接编辑能力

缺点：

- 需要引入 Tab 状态模型与后端持久化能力
- 需要重新组织现有查询中心子页面

### 方案 C：查询资产页 + 编辑器混合

做法：

- 保留 `我的查询 / 历史 / 模板 / 定时查询` 的相对独立页面
- `/queries` 只新增多 Tab 编辑器

优点：

- 迁移阻力最低

缺点：

- 入口仍然分散
- 用户心智仍会在多个页面之间来回切换
- 不符合 IDE 式工作台的整体目标

### 推荐结论

采用 **方案 B：IDE 式查询工作台**。

## 5. 信息架构

### 5.1 主入口与职责

#### `/queries`

职责：

- 多 Tab SQL 编辑与执行
- 数据源绑定
- 查询保存与更新
- 基于查询发布任务
- 从资源、我的查询、模板、历史中打开查询上下文

不承担：

- 复杂调度管理
- 任务运行监控总览

#### `/queries/scheduled`

职责：

- 已发布查询任务的管理
- 启停任务
- 立即执行
- 查看任务状态与运行入口

不承担：

- SQL 编辑
- 模板应用
- 历史回放主流程

### 5.2 查询工作台骨架

查询工作台采用四层结构：

1. 顶部 Tab 栏
2. 顶部工具栏
3. 左侧资产与资源面板
4. 中央编辑区 + 下方结果区

#### 第一层：Tab 栏

每个 Tab 表示一个独立查询上下文，包含：

- Tab 标题
- 未保存状态
- 当前数据源标识
- 已保存查询 ID（若有）

操作支持：

- 新建 Tab
- 切换 Tab
- 关闭 Tab
- 重命名 Tab（由保存查询名称驱动）

#### 第二层：工具栏

固定核心动作：

- `运行`
- `停止`
- `格式化`
- `保存查询`
- `发布为任务`

其中：

- `发布为任务` 仅在当前 Tab 已保存为查询后可用
- 删除当前“推荐路径”提示条，不保留单独说明区

#### 第三层：左侧面板

左侧统一为一个可切换面板，包含四个分区：

- `资源`
- `我的查询`
- `模板`
- `历史`

不再为这四类对象分别维持强主路径页面。

#### 第四层：中央编辑区与结果区

- 中央区域：SQL 编辑器
- 下方区域：当前 Tab 的结果 / 错误 / 运行信息

结果区只属于当前 Tab，不与其他 Tab 共享。

## 6. Tab 状态模型

### 6.1 单 Tab 的状态内容

每个 Tab 至少维护以下字段：

- `tab_id`
- `title`
- `source_id`
- `sql_query`
- `query_id`
- `query_name`
- `is_dirty`
- `last_saved_at`
- `result_snapshot`
- `error_snapshot`
- `execution_status`
- `opened_from`

其中：

- `query_id` 表示是否已经保存为查询资产
- `is_dirty` 表示当前编辑内容是否与已保存状态不一致
- `result_snapshot / error_snapshot` 只属于该 Tab

### 6.2 持久化要求

由于用户已明确要求采用“像 IDE 一样持久”，因此 Tab 需要后端持久化：

- 刷新页面后恢复 Tab 列表
- 跨设备恢复最近的工作台上下文
- 恢复当前激活 Tab
- 恢复每个 Tab 的数据源、SQL 草稿和已保存查询 ID

本轮设计上不允许退化为“仅浏览器本地缓存”。

### 6.3 Tab 行为规则

#### 新建 Tab

- 默认标题：`未命名查询`
- 初始 SQL：空白或示例 SQL
- 默认未保存

#### 切换 Tab

- 切换后恢复该 Tab 对应的数据源、SQL、结果和状态
- 不共享执行结果

#### 关闭 Tab

- 若无未保存改动，直接关闭
- 若存在未保存改动，弹出确认：
  - `保存后关闭`
  - `直接关闭`
  - `取消`

## 7. 查询资产与任务的事件流

### 7.1 保存查询

点击 `保存查询` 时：

- 若当前 Tab 尚未保存：
  - 弹窗收集 `查询名称 / 描述 / 标签`
  - 保存成功后生成 `query_id`
  - Tab 标题切换为查询名称
- 若当前 Tab 已保存：
  - 直接更新查询资产

此时保存的是 **查询资产**，不是任务实体。

### 7.2 发布为任务

点击 `发布为任务` 时：

- 若当前 Tab 未保存为查询：
  - 提示 `请先保存查询`
  - 不允许直接发布
- 若当前 Tab 已保存：
  - 弹出轻量发布弹窗
  - 配置最小必要字段：
    - 任务名称
    - 任务描述
    - 执行方式：`手动执行 / 定时执行`

发布成功后：

- 生成任务实体
- 提供两个后续动作：
  - `查看任务`
  - `继续留在查询工作台`

### 7.3 事件流总览

主链路固定为：

`新建 Tab -> 编辑 SQL -> 运行验证 -> 保存查询 -> 发布为任务`

而不是：

`写 SQL -> 直接变成任务`

## 8. 子页面收口策略

### 8.1 我的查询

- 不再承载主编辑工作流
- 并入 `/queries` 左侧的 `我的查询`
- 点击某个查询时：
  - 在当前工作台打开一个新 Tab
  - 不再跳出到独立编辑页

### 8.2 历史

- 并入 `/queries` 左侧的 `历史`
- 点击历史记录时：
  - 用对应 SQL 和数据源打开新 Tab
  - 可以带入历史来源标识

### 8.3 模板

- 并入 `/queries` 左侧的 `模板`
- 支持两种动作：
  - 应用到当前 Tab
  - 新开 Tab 使用模板

### 8.4 定时查询

- 保留 `/queries/scheduled`
- 语义改为：`查询任务管理页`
- 不再被理解为 SQL 编辑工作区

### 8.5 兼容路由

旧入口继续允许访问，但统一回流到 `/queries` 并带兼容参数，例如：

- `/queries/my`
- `/queries/history`
- `/queries/templates`

这些路径不再作为长期主入口，只承担兼容职责。

## 9. 错误处理

### 9.1 未保存状态

- Tab 有未保存改动时，标题显示未保存标记
- 离开页面或关闭 Tab 时统一走拦截确认

### 9.2 发布前置校验

- 未保存查询禁止发布任务
- 数据源缺失时禁止执行
- SQL 为空时禁止执行与保存

### 9.3 执行错误

- 错误只显示在当前 Tab 结果区
- 不跨 Tab 污染执行状态

## 10. 测试与验收

### 10.1 核心验收项

1. `/queries` 支持同时打开多个查询 Tab
2. 每个 Tab 独立绑定数据源、SQL、结果和保存状态
3. 刷新页面后可恢复最近打开的 Tab 与激活态
4. 未保存查询不能直接发布为任务
5. 已保存查询可以发布为任务
6. `我的查询 / 历史 / 模板` 在工作台内即可打开查询上下文
7. `/queries/scheduled` 仍作为任务管理页可用

### 10.2 推荐测试层次

- 单元测试：Tab 状态管理、关闭确认、发布前置校验
- 页面测试：多 Tab 新建/切换/关闭/恢复、保存查询、发布任务弹窗
- 集成测试：`/queries` 与 `/queries/scheduled` 的主链路协同

## 11. 实施顺序

1. 先重构 `/queries` 的前端状态模型，引入多 Tab
2. 收口左侧面板，将 `我的查询 / 历史 / 模板` 并入工作台
3. 接入查询持久化恢复
4. 接入 `保存查询 -> 发布为任务`
5. 最后保留并收口 `/queries/scheduled` 的任务管理语义

## 12. 结论

本设计将查询中心收口为 **IDE 式多 Tab SQL 工作台**，核心心智如下：

- `Tab` 是查询开发上下文
- `查询` 是可保存、可复用的资产
- `任务` 是由查询发布得到的运行实体

这样既能满足用户对 SQLLab 的预期，也能保持平台边界清晰，符合 `KISS / YAGNI / SOLID / DRY` 四项原则。
