# 评审规则

本文档定义仓库的评审规则与拒绝条件。
它回答的是“改动在测试通过后，是否仍然不应合并”。

`testing` 证明的是“当前没有明显坏掉”，`review` 判断的是“这次改动是否合适、是否值得接受”。
因此，`make review` 只是收集验证证据的统一入口，不等于评审结论本身。

## 1. 职责边界

- `AGENTS.md`：只引用评审规则，不维护拒绝条件明细。
- `docs/quality/testing.md`：定义验证入口、分层矩阵和变更范围到验证要求的规则。
- `docs/quality/review.md`：定义评审时的拒绝条件和补充判断标准。
- `make review`：执行仓库级验证、文档健康检查和文档影响检查，给评审提供最小证据包；它不替代人工或 agent 的质量判断。

## 2. 评审输入

评审至少应基于以下输入判断：

- 任务目标与原始约束
- 代码差异和受影响文档
- `make verify-*`、`make verify` 或 `make review` 的结果
- 运行时验证、专项 smoke 或回归验证结果
- 已知风险、未覆盖项和明确接受的例外

如果这些输入不完整，结论只能是“暂不通过”，而不是“默认接受”。

## 3. 拒绝条件

出现以下任一情况时，不应合并：

- 改动没有直接完成任务目标，或只完成了部分关键路径。
- 改动范围明显超出任务目标，并引入无关的大面积重写、迁移或结构调整。
- 引入重复实现、旁路现有抽象、绕开统一入口或破坏既有分层边界。
- 改变了用户可见行为、接口语义、配置方式、运行方式或工作流，但未同步更新相关文档。
- 必需验证未运行、运行失败、结果无法解释，或仅以局部测试替代要求的交付入口。
- 改动影响运行时行为，却没有 smoke 或等效运行时证据。
- 引入已知兼容性风险、关键链路回归、安全问题、性能退化或稳定性风险，但没有明确缓解措施或接受说明。
- API、Schema、共享契约或跨端数据结构发生变化，但没有补充兼容性、契约或跨域验证。
- 构建、工具链或配置改动使默认开发流程不可用，且未提供迁移说明。
- 将临时调试代码、测试资产、生成物或状态性副作用错误地带入基线，且未说明其必要性。
- 代码虽然能跑通，但复杂度明显高于需求，违反 KISS / YAGNI，后续维护成本不成比例。

## 4. 补充判断标准

在没有触发硬性拒绝条件时，再判断以下问题：

- 是否复用了现有模块边界，而不是新开一条平行路径。
- 是否把高风险变化限制在最小改动面内。
- 是否为后续维护者保留了足够清晰的入口、约束和文档。
- 是否把“当前需要”与“未来可能需要”分开，避免无谓预埋。

这些判断用于决定“是否值得接受”，不是替代前面的拒绝条件。

## 5. 与 `make review` 的关系

`make review` 当前执行：

- `make verify`
- `make verify-docs`
- `make docs-impact`

这意味着它提供的是：

- 仓库级四层验证证据
- 文档健康检查证据
- 文档影响检查证据

但以下判断仍然需要 `review.md` 负责：

- 任务是否真正完成
- 改动是否过度设计
- 是否引入了不应接受的范围外变更
- 文档是否描述了正确的行为和约束
- 风险是否已被解释、缓解或明确接受

## 6. 使用建议

推荐顺序：

1. 先根据 `docs/quality/testing.md` 跑完所需验证
2. 需要仓库级证据时再运行 `make review`
3. 最后用本文件的拒绝条件判断“是否应合并”

如果无法明确回答“为什么这次改动应该被接受”，就不应把结论写成“可合并”。
