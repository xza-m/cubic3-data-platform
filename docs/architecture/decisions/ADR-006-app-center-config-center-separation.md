# ADR-006 应用中心与配置中心保持职责分离，通过实例与订阅关联

## 状态

当前有效

## 背景

当前平台在路由、导航和 API 上已经把“应用”与“投递配置”拆成两个中心：

- 应用中心
  - `/apps`
  - `/executions`
  - `/api/v1/apps`
  - `/api/v1/app-instances`
  - `/api/v1/app-executions`
- 配置中心
  - `/config/channels`
  - `/config/subscriptions`
  - `/api/v1/channels`
  - `/api/v1/subscriptions`

其中订阅与应用实例存在关联，但订阅本身并不等于应用实例，也不应并回应用定义或执行记录。

## 决策

当前平台保持以下边界：

- 应用中心负责“要运行什么”和“运行了什么”
  - 运行型应用定义
  - 应用实例
  - 执行记录
- 交互型问答能力优先进入智能问数域，不进入应用中心主 IA
  - Web chatbot
  - 飞书问数 Bot
- `DataAgent` 作为运行型 Agent 应用继续归入应用中心的运行域管理；其问数能力不作为应用中心一级入口对外呈现
- 配置中心负责“通过什么渠道、按什么规则投递”
  - 渠道
  - 订阅
  - 投递过滤与交付配置

两者通过 `app_instance_id` 等关联字段连接，并允许提供便捷查询，例如 `/api/v1/app-instances/<id>/subscriptions`，但不合并为单一中心。

## 理由

- 应用定义/实例/执行属于运行域，渠道/订阅属于投递配置域，职责不同
- “一次触发一次执行”的运行型能力与“多轮会话”的交互型能力对象模型不同，不应强行共模
- 把投递配置从应用域中拆开，有利于复用渠道和订阅规则
- 前端导航分成应用中心与配置中心后，用户心智更清楚：一个负责“执行”，一个负责“连接和订阅”

## 结果与约束

正面结果：

- 执行链路与交付链路职责更清楚
- 渠道和订阅可独立治理，不被某个应用域绑死
- 前后端都可以围绕同一分工继续扩展

约束：

- 不要把渠道配置直接塞进应用定义或实例作为主模型
- 不要把订阅管理退化成应用执行的附属表单
- 新的交付规则、通知策略优先归入配置中心，而不是应用中心
- 如果需要跨中心展示，应通过关联查询和聚合视图实现，而不是打破领域边界
