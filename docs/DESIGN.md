# 设计语言基线

本文档定义当前项目的统一 UI/UX 设计语言。
它不是单个页面的视觉说明，而是整个平台的界面风格基线，供前端实现、Pencil 设计稿、评审与后续改版共同遵循。

当前推荐的统一标注为：

**Enterprise SaaS Workbench / Data Console UI**

也可用中文表述为：

**企业级数据平台的工作台式后台设计**

## 1. 设计定位

项目整体界面不采用营销型 SaaS、内容消费型产品或文档优先产品的设计范式，而采用以下组合范式：

- 信息密集型后台
- IDE / Explorer 式信息架构
- Master-detail 三栏工作台
- 功能优先的企业级设计系统

一句话总结：

这是一个以效率、结构、对象管理和连续操作为优先级的专业工作平台，而不是强调品牌情绪和装饰体验的展示型产品。

## 2. 核心设计原则

### 2.1 信息密度优先

页面优先展示结构化信息、对象关系和操作入口，而不是通过大面积留白或装饰性区块制造“轻盈感”。

体现方式：

- 一屏内尽可能展示更多有效信息
- 主要载体是表格、列表、属性面板、Tab、树结构
- 通过间距、边框、字重建立层级，而不是依赖大色块

### 2.2 工作台优先

页面应像工作台，而不是像宣传页。

体现方式：

- 左侧是全局导航或资源树
- 中间是主工作区
- 右侧是属性、状态或上下文检查器
- 顶部保留搜索、状态、主动作和模式切换

### 2.3 结构强于装饰

界面层级主要依靠布局、分栏、标题系统、表格列宽、边框和背景层级建立。

避免：

- 大面积 Hero
- 无业务意义的装饰插画
- 过多玻璃态、渐变、投影特效
- 为了“好看”牺牲信息密度

### 2.4 连续操作优先

用户在同一页面中应能完成浏览、选择、编辑、预览、保存、切换上下文，而不是频繁跳页。

体现方式：

- 对象列表与详情联动
- 就地切换 Tab
- 右键菜单、更多菜单、Inspector 面板承载局部操作
- 预览、代码、属性编辑尽量处于同一工作上下文

### 2.5 专业可信

设计气质应传达“稳定、可靠、专业、可控”。

因此：

- 色彩以中性、低饱和为主
- 状态色只表达状态，不承载品牌情绪
- 文案直接、清楚、少夸张表达
- 数字、状态、对象关系应明确可读

## 3. 设计范式

### 3.1 Information-dense Admin Console

这是项目的第一层范式。

典型结构：

- 左侧全局导航
- 顶部全局搜索或全局工具栏
- 主区域以表格、对象列表、统计摘要、操作区为主
- 很少使用营销化视觉语言

参考类型：

- Databricks
- Snowflake
- AWS / GCP 控制台
- GitHub 企业后台
- BI、数据平台、云平台控制台

### 3.2 IDE / Explorer 式布局

这是项目在数据资产、语义建模、对象管理场景中的第二层范式。

典型结构：

- 左侧资源树、Catalog 树或对象浏览器
- 中间主视图展示当前对象
- 顶部使用 breadcrumb、Tab、模式切换
- 右侧属性区承担 metadata / status / 编辑职责

这使数据资产管理更接近开发工具体验，而不是传统业务后台。

### 3.3 Master-detail 三栏工作台

这是工作台类页面的首选布局模式。

- 左栏：对象集合、目录树、资源筛选
- 中栏：当前对象主体内容
- 右栏：对象属性、状态、检查器、上下文动作

适用页面：

- 语义工作台
- 业务语义工作台
- 领域建模与关系画布
- 数据资产浏览与对象详情页

## 4. 视觉语言

### 4.1 色彩

整体采用浅色、低饱和、中性偏冷的配色体系。

基线原则：

- 主背景以白色和浅灰色为主
- 主强调色使用冷蓝色系
- 成功、警告、错误使用标准状态色
- 大面积高饱和品牌色不是主要视觉手段

设计角色：

- 白色：主工作表面
- 浅灰：分区、弱背景、层级区分
- 蓝色：当前选中、主动作、链接、强调信息
- 绿色 / 橙色 / 红色：状态反馈

### 4.2 层次

采用“弱层次、强结构”。

具体表现：

- 用 1px 边框区分层级
- 用轻背景差异区分区域
- 用字号、字重和行高形成阅读秩序
- 阴影仅作轻微浮起，不做主视觉

### 4.3 圆角与控件形态

整体圆角为中等圆角，保持现代感，但避免过度柔软。

建议：

- 工作区卡片和面板：中等圆角
- 输入框、按钮、chip：小到中等圆角
- 不使用极大圆角营造消费级气质

### 4.4 图标

图标应以功能表达为主，不应成为装饰中心。

要求：

- 线性图标优先
- 尺寸统一
- 在导航、表格、字段类型、状态标识中承担信息编码作用

## 5. 排版语言

### 5.1 标题系统

标题要明确表达对象层级，而不是追求视觉戏剧性。

建议层级：

- 页面标题：对象名、当前工作台名
- 区块标题：列表区、属性区、预览区
- 表头与字段标签：小而稳定

标题应更紧凑、更克制，适合工作台密集阅读。

### 5.2 正文字体

正文应优先选择适合高频后台阅读的无衬线字体系统。

要求：

- 中英文混排清晰
- 数字与字段名可读性强
- 技术字段、代码、表达式使用等宽字体

### 5.3 数字与代码

涉及以下内容时优先使用更稳定的数字与等宽风格：

- SQL
- YAML
- 表达式
- ID、时间、统计数字
- 列名、字段名、枚举值

## 6. 布局规范

### 6.1 全局壳层

平台默认采用：

- 固定左侧导航
- 顶部辅助工具栏或页面头部
- 主区域自适应内容区

左侧导航承担全局信息架构，不应频繁变化风格。

### 6.2 页面布局

优先使用以下布局模型：

1. 控制台首页：统计 + 快捷入口 + 最近活动
2. 列表页：筛选栏 + 表格 + 行操作
3. 工作台页：三栏布局
4. 详情页：主内容 + 右侧属性摘要

### 6.3 页面头部

页面头部只承载必要信息：

- 页面名 / 对象名
- breadcrumb
- 状态
- 主动作
- 模式切换

不应在头部加入大面积介绍文案或装饰区。

## 7. 核心组件语汇

以下组件是本项目设计语言中的高频基础单元：

- Sidebar：全局导航
- Resource Tree：资源树 / Catalog 树
- Table：数据表格、对象列表
- Inspector：属性面板 / 状态面板
- Tabs：工作模式与对象分区切换
- Toolbar：页面头部操作区
- Badge：状态、类型、标签
- Context Menu / More Menu：次级操作
- Search Input：全局搜索、局部筛选

它们共同构成“工具型界面语言”。

## 8. 页面行为规范

### 8.1 就地编辑

如果用户已经进入某对象的工作上下文，应优先提供就地编辑，而不是跳往新页面。

### 8.2 局部显现

次级操作默认不抢主视觉，可在以下时机显现：

- hover
- 右键
- 行尾 `...`
- Inspector 选中态

### 8.3 高级能力渐进暴露

高级能力可以保留，但默认不应压过主任务。

例如：

- YAML / Python
- SQL 预览结果
- 调试信息
- 关系画布

应通过 Tab、抽屉、次级模式切换暴露，而不是全量平铺。

### 8.4 空状态

空状态应服务于“下一步行动”，而不是表达品牌感。

空状态应回答三个问题：

- 当前为什么是空的
- 用户下一步该做什么
- 从哪里开始

## 9. 什么不属于本项目风格

以下方向不属于当前统一设计语言，除非是专题页面且明确批准：

- 营销型 SaaS Landing Page 风格
- 大面积渐变、光效、玻璃态主导的界面
- 消费级 App 风格的大图标、大留白
- Notion 式文档优先布局
- Linear 式极窄极简协作工具风格
- Apple 式内容型应用语言

换句话说，本项目不追求“情绪化高级感”，而追求“专业、清晰、稳定、可操作”。

## 10. 语义工作台专项要求

语义工作台是本项目最典型的设计语言承载页，应遵循以下专项规则：

- 页面必须优先体现工作台属性，而不是编辑器壳层拼贴
- 左栏承担资源与字段浏览
- 中栏承担主建模任务
- 右栏承担属性、状态和对象检查
- 表格、字段列表、Inspector 必须是主角色
- 高级能力如 YAML、Python、Preview 应以模式切换或次级区域呈现
- 可以参考 Databricks 的结构感，但不能照搬品牌视觉

## 11. 实现约束

从工程原则看，设计语言落地也应遵循：

- `KISS`：优先使用稳定分栏、表格、面板，不引入无意义视觉复杂度
- `YAGNI`：不为未来假设场景堆叠多套 UI 模式
- `SOLID`：导航、列表、详情、Inspector 职责清晰
- `DRY`：颜色、圆角、间距、阴影、字体尽量通过 token 和共享组件统一
