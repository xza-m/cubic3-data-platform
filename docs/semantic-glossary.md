# 语义层统一术语表

本文档用于统一语义层相关术语，降低跨团队沟通成本和新同学 onboarding 成本。
除架构实现、代码目录和 ADR 外，产品说明、需求讨论、页面文案和跨团队沟通默认以本文档术语为准。

## 0. 3 分钟速读版

如果你是首次接触语义层，先记住下面 5 条：

1. 语义层对外只保留 3 个一级概念：`业务语义层`、`分析模型层`、`语义执行层`。
2. `业务语义层` 负责定义业务上“是什么”，例如业务对象、业务指标、术语和语义权限。
3. `分析模型层` 负责定义技术上“怎么算”，例如 Cube、维度、度量、Join 和分析逻辑。
4. `语义执行层` 负责把用户问题转成执行路径，包括语义路由、执行规划、执行预览和运行时执行。
5. `Semantic Mapper` 和 `Execution Compiler` 仍然存在于内部实现中，但默认不作为跨团队一级概念对外沟通；对应统一说法分别是 `对齐检查/影响分析` 与 `执行预览/执行引擎`。

最短记忆版：

- 业务语义层：定义“是什么”
- 分析模型层：定义“怎么算”
- 语义执行层：负责“怎么执行”

如果只需要理解主线，读完本节即可；如果需要参与需求讨论、设计评审或页面文案收敛，再继续阅读后文。

## 1. 总原则

### 1.1 对外只保留 3 个一级概念

- `业务语义层`
- `分析模型层`
- `语义执行层`

### 1.2 不再作为一级概念对外频繁使用的词

- `Semantic Mapper`
- `Execution Compiler`

它们仍然是当前实现中的内部模块，但在多数产品说明、需求讨论和页面文案中，应改用能力描述词，而不是系统层名。

### 1.3 术语收敛目标

- 降低概念数量，符合 `KISS`
- 避免同一职责被多套名词描述，符合 `DRY`
- 保留业务语义与执行语义分层，符合 `SOLID`
- 不过早把内部实现产品化，符合 `YAGNI`

## 2. 一级概念

### 2.1 业务语义层

定义业务世界中的对象、属性、关系、动作、业务指标和语义权限。

回答的问题：

- 业务上“这是什么”
- 业务口径“怎么算”
- 哪些角色“能不能看”

包含内容：

- 业务对象
- 业务属性
- 业务关系
- 业务动作
- 业务指标
- 术语
- 语义权限

推荐说法：

- “这个需求先在业务语义层定义口径”
- “这个业务指标属于业务语义层资产”

避免说法：

- “先去 Ontology 那边改一下”
- “这是 Ontology 的东西，不是业务语义”

### 2.2 分析模型层

定义可执行的分析实体、维度、度量、Join 和分析逻辑。

回答的问题：

- 技术上“这怎么查”
- 指标“怎么执行”
- 维度和关联“怎么落地”

包含内容：

- Cube
- Measure
- Dimension
- Join
- View
- Recipe

推荐说法：

- “这个逻辑需要在分析模型层落地”
- “这个业务指标当前还没有对应的分析模型实现”

避免说法：

- “去 Cube 那边改一下就行”
- “这是语义层，不是模型层”

### 2.3 语义执行层

把用户问题转换成可执行路径，并完成预览、执行和回溯。

回答的问题：

- 这个问题该怎么执行
- 应该走 SQL、检索还是工具
- 执行计划是什么
- 执行结果和治理结果是什么

包含内容：

- 语义路由
- 执行规划
- 执行预览
- 运行时执行
- 治理回溯

推荐说法：

- “这个问题已经进入语义执行层”
- “先看执行预览，再决定是否执行”

避免说法：

- “走 Router 还是走 Compiler”
- “这个请求是 Compiler 处理的，不是执行层处理的”

## 3. 二级术语

### 3.1 业务对象

业务上独立存在、可被识别和分析的核心实体，例如订单、学生、课程、班级。

归属层级：

- 业务语义层

不要混淆：

- 业务对象 ≠ 数据表
- 业务对象 ≠ Cube

### 3.2 业务属性

业务对象的特征或字段语义，例如订单状态、支付时间、区域、学生年级。

归属层级：

- 业务语义层

不要混淆：

- 业务属性 ≠ 物理字段
- 业务属性 ≠ 维度，但常常会映射到维度

### 3.3 业务关系

两个业务对象之间的业务关联，例如订单属于用户、学生属于班级。

归属层级：

- 业务语义层

### 3.4 业务动作

业务对象发生的业务事件或动作，例如下单、支付、提交作业、完成课程。

归属层级：

- 业务语义层

### 3.5 业务指标

用业务口径表达的指标定义，不直接承载 SQL，例如 GMV、转化率、学习完成率。

归属层级：

- 业务语义层

关键原则：

- 业务指标表达“业务口径”
- 不直接表达“执行公式”

不要混淆：

- 业务指标 ≠ Measure
- 业务指标 ≠ SQL 指标

### 3.6 分析实体

可参与分析执行的模型实体，通常由 Cube 体系承载。

归属层级：

- 分析模型层

推荐用途：

- 面向产品和业务沟通时使用“分析实体”
- 面向模型开发沟通时可继续使用 Cube

### 3.7 分析指标

可直接参与执行计算的指标定义，例如 Measure、Derived Metric。

归属层级：

- 分析模型层

不要混淆：

- 分析指标负责“怎么计算”
- 业务指标负责“业务上是什么意思”

### 3.8 语义权限

挂在业务对象、动作、业务指标上的最小权限元数据。

归属层级：

- 定义上属于业务语义层
- 生效上属于语义执行层

推荐说法：

- “这个指标配置了语义权限”
- “该角色在执行层被权限阻断”

### 3.9 对齐检查

检查业务语义层和分析模型层是否对齐的能力。

来源：

- 当前内部实现可对应 `Semantic Mapper`

推荐说法：

- “先做对齐检查”
- “这个业务指标还没有完成模型对齐”

避免说法：

- “Mapper 过了吗”

### 3.10 影响分析

分析某个业务语义资产变化后会影响哪些模型、查询和执行链路。

推荐说法：

- “发布前先看影响分析”
- “这个变更会影响 3 个分析实体”

### 3.11 执行预览

对目标执行路径进行预览，查看可能生成的 SQL、检索计划或工具调用。

来源：

- 当前内部实现可对应 `Execution Compiler Preview`

推荐说法：

- “先看执行预览”
- “这个业务指标已经具备执行预览能力”

避免说法：

- “Compiler 能不能编过去”

### 3.12 治理回溯

查看执行时命中的权限、决策、审计记录和执行状态。

推荐说法：

- “去看治理回溯”
- “这次执行有审计记录可查”

## 4. 必须区分的概念对

### 4.1 业务对象 vs 分析实体

- 业务对象：业务世界里的核心概念
- 分析实体：执行分析用的模型实体

一句话记忆：

前者面向业务理解，后者面向执行落地。

### 4.2 业务指标 vs 分析指标

- 业务指标：表达业务口径
- 分析指标：表达执行计算

一句话记忆：

前者回答“是什么”，后者回答“怎么算”。

### 4.3 对齐检查 vs 主定义

- 对齐检查：校验两层是否一致
- 主定义：真正承载真相的资产定义

一句话记忆：

对齐检查不是第三套真相源。

### 4.4 执行预览 vs 运行时执行

- 执行预览：先看计划
- 运行时执行：真正执行

一句话记忆：

预览是验证，执行是落地。

## 5. 推荐统一表达

### 5.1 一句话介绍

平台语义层采用三层表达：业务语义层定义业务口径，分析模型层定义执行逻辑，语义执行层负责把问题转成可执行结果。

### 5.2 需求讨论说法

- “这个需求先补业务语义定义，再确认分析模型映射。”
- “这不是业务语义问题，是分析模型缺实现。”
- “这个问题已经能走语义执行链，但还需要权限校验。”
- “先看对齐检查和影响分析，再决定是否发布。”

## 6. 不推荐的表达

### 6.1 不推荐频繁使用的词

- `Ontology`
- `Semantic Mapper`
- `Execution Compiler`
- `Policy Metadata`

这些词可以在架构实现、代码层和 ADR 中保留，但不建议作为跨团队主沟通词。

### 6.2 推荐替换

- `Ontology` -> `业务语义`
- `Semantic Mapper` -> `对齐检查`
- `Execution Compiler` -> `执行预览` 或 `执行引擎`
- `Policy Metadata` -> `语义权限`

## 7. 团队统一口径结论

### 7.1 保留为一级概念

- 业务语义层
- 分析模型层
- 语义执行层

### 7.2 降级为能力名

- 对齐检查
- 影响分析
- 执行预览
- 治理回溯

### 7.3 不应再被误认为一级系统层

- `Semantic Mapper`
- `Execution Compiler`

## 8. 新同学 onboarding 最短记忆版

只记住这三句话即可：

1. `业务语义层` 定义业务上“是什么”。
2. `分析模型层` 定义技术上“怎么算”。
3. `语义执行层` 负责把问题“变成可执行结果”。
